Модуль E — PBIP / PBIR

Що таке PBIP

PBIP — збереження звіту й моделі не одним бінарним файлом, а папкою текстових і JSON-файлів. Саме це робить Power BI сумісним із Git.

Ціль урокупояснити, чому PBIP-формат робить Power BI сумісним із Git.
Бінарний vs текстовий файл
Бінарний файл (як .pbix) — суцільний нечитабельний блок байтів. Текстовий — рядки, які можна читати й порівнювати.
Простими словами: бінарник — як запечатана коробка (видно лише «змінилась/ні»). Текст — як відкритий зошит (видно кожне слово).
PBIP (Power BI Project)
Формат, у якому Power BI Desktop зберігає проєкт як папку текстових файлів замість одного .pbix.
Простими словами: «розпакований» Power BI, де модель і звіт лежать окремими читабельними файлами.

.pbix

Один бінарний файл. Git бачить як суцільний блок: діфи, гілкування, review неможливі.

PBIP

Папка текстових файлів. Кожна міра, таблиця, візуал — окремо й читабельно. Git показує точні зміни.

Як зберегти

Power BI Desktop → File → Save As → «Power BI project files (*.pbip)». Desktop створить структуру папок і .gitignore автоматично.

Стан на 2026. Звітна частина PBIP за замовчуванням використовує формат PBIR (нова модульна структура). .pbix лишається основним контейнером і нікуди не зникає — він теж усередині перейшов на PBIR-метадані.

Модуль E — PBIP / PBIR

Формати всередині: PBIR та TMDL

PBIP — це контейнер. Звіт усередині описано форматом PBIR, модель — мовою TMDL.

Ціль урокучитати формати сторінок звіту (PBIR) і моделі (TMDL).
Семантична модель vs звіт
Модель — таблиці, зв'язки, міри (логіка й дані). Звіт — сторінки з візуалами, що показують модель.
Простими словами: модель — це «кухня й рецепти», звіт — «подача страви на тарілці».

TMDL — мова опису моделі

TMDL (Tabular Model Definition Language)
Текстова мова опису моделі з відступами замість дужок. Людиночитна, чудово діфиться.
Простими словами: «модель словами»: рядок за рядком описано кожну таблицю, колонку й міру.
table Sales
    measure 'Total Sales' = SUM(Sales[Amount])
        formatString: #,0
        lineageTag: 3f9a-...-c21

    column Amount
        dataType: double

PBIR — формат звіту

PBIR (Power BI Enhanced Report Format)
Модульний JSON-опис звіту: сторінки, візуали, закладки — кожне в окремому файлі в definition/.
Простими словами: звіт «по деталях»: кожен візуал — окремий файл, тому Git точно бачить, що саме змінилось.
lineageTag
Унікальний GUID, присвоєний кожному об'єкту моделі. Стабілізує його ідентичність між середовищами.
Простими словами: «незмінний серійний номер» міри чи таблиці. Паралельна робота двох гілок робить його типовим джерелом конфліктів (модуль F).
Інструмент. TMDL зручно редагувати у VS Code (IntelliSense) або Tabular Editor; PBIR JSON — у VS Code з його схемами для перевірки.
Модуль E — PBIP / PBIR

Налаштування PBIP-репозиторію з нуля

Покрокова послідовність: від .pbix до спільного репозиторію на сервері.

Ціль урокупровести проєкт від збереження як PBIP до першого push.

Крок 1 — зберегти як PBIP

У Power BI Desktop: File → Save As → «Power BI project files (*.pbip)», обери папку. Desktop створить структуру й .gitignore.

Крок 2 — увімкнути Git у папці

cd C:\Repos\PDP          # перейти в папку проєкту
git init                  # увімкнути контроль версій
git add .                 # усі файли проєкту у staging
git commit -m "Початковий стан PDP"

Крок 3 — під'єднати сервер і відправити

Що відбувається: локально → сервер
git remote add origin <url>   # під'єднати сервер (Azure DevOps / GitHub)
git push -u origin main       # відправити перший раз

Крок 4 — колега забирає копію

git clone <url>            # колега завантажує проєкт собі
Перевірка перед першим коммітом: виконай git status і переконайся, що cache.abf та localSettings.json НЕ потрапили у список (їх має ховати .gitignore). Якщо вони видні — див. модуль F, кейс про git rm --cached.
Модуль E — PBIP / PBIR

Обмеження PBIP та коли його не варто застосовувати

Чесний погляд: PBIP чудовий для контролю версій, але має межі.

Ціль урокузнати межі формату і способи їх обходити.

Обмеження

  • Великі моделі. Дуже великі семантичні моделі дають важкі TMDL-діфи; review ускладнюється.
  • Складні merge. Текстовий merge може дати синтаксично валідний, але логічно зламаний файл (дубль lineageTag, биті зв'язки) — потрібна валідація в Desktop / Tabular Editor.
  • Не все у звіті. Окремі специфічні можливості звіту можуть бути представлені неповно; перевіряй після злиття.
  • Бінарні залишки. Зображення та інші asset-и лишаються бінарними — Git їх не діфить (для великих → Git LFS).
  • Дані поза Git. Дані в репозиторії не зберігаються; партиції / incremental refresh живуть у Service і перебудовуються після деплою.

Коли краще НЕ покладатися лише на PBIP

  • Дуже велика модель з інтенсивними паралельними правками одних і тих самих таблиць → розглянути роботу через Tabular Editor (стек 2) з чітким розподілом зон.
  • Команда без дисципліни короткоживучих гілок → конфлікти накопичуються; спершу впорядкувати процес.
Висновок. PBIP лишається основою контролю версій для Power BI; обмеження вирішуються дисципліною (малі гілки, валідація після merge, .gitattributes, LFS для бінарників) та, за потреби, стеком Tabular Editor.
Модуль E — PBIP / PBIR

Стек 2: Semantic Model + Tabular Editor

Альтернатива до «PBIP + Power BI Desktop»: модель редагують у Tabular Editor, який дає точніший контроль і автоматичну перевірку якості.

Ціль урокуредагувати модель у Tabular Editor і перевіряти її BPA.
Tabular Editor
Окремий інструмент для редагування семантичної моделі (таблиць, мір, зв'язків) на рівні TMDL, поза Power BI Desktop.
Простими словами: «редактор моделі під капотом» — швидший і точніший для масових змін мір та метаданих, ніж Desktop.
Best Practice Analyzer (BPA)
Вбудований у Tabular Editor аналізатор, що перевіряє модель за набором правил (іменування, описи, формати, продуктивність).
Простими словами: «лінтер моделі» — підсвічує порушення правил ще до коміту чи деплою.

Робочий цикл стеку 2

git pull                          # свіжий стан
# File → Open → Model from Folder → Model/   (відкрити TMDL-папку в Tabular Editor)
# внести зміни: Measures → New Measure → DAX
# Tools → Best Practice Analyzer → виправити зауваження
# Ctrl+S → зміни записуються у .tmdl файли
git add . && git commit -m "feat(model): YTD Sales measure"
git push                          # далі — Git sync у Dev → deployment pipeline у Prod
Стек 1PBIP + Power BI Desktop. Модель і звіт разом; зручно для звітної частини й загальної розробки.
Стек 2Semantic Model + Tabular Editor. Точний контроль моделі, BPA, масові правки мір; звіт — окремо.
На практиці стеки поєднують: звіт правлять у Desktop (PBIP), а складні зміни моделі — у Tabular Editor. Обидва пишуть ті самі .tmdl, тож Git і деплой однакові.
