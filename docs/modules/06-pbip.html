Модуль E — PBIP / PBIR

Що таке PBIP

PBIP — збереження звіту й моделі не одним бінарним файлом, а папкою текстових і JSON-файлів. Саме це робить Power BI сумісним із Git.

Ціль урокупояснити, чому PBIP-формат робить Power BI сумісним із Git.
Бінарний vs текстовий файл
Бінарний файл (як .pbix) — суцільний нечитабельний блок байтів. Текстовий — рядки, які можна читати й порівнювати.
Простими словами: бінарник — як запечатана коробка (видно лише «змінилась/ні»). Текст — як відкритий зошит (видно кожне слово).
PBIP (Power BI Project)
Формат, у якому Power BI Desktop зберігає проєкт як папку текстових файлів замість одного .pbix.
Простими словами: «розпакований» Power BI, де модель і звіт лежать окремими читабельними файлами.

.pbix

Один бінарний файл. Git бачить як суцільний блок: діфи, гілкування, review неможливі.

PBIP

Папка текстових файлів. Кожна міра, таблиця, візуал — окремо й читабельно. Git показує точні зміни.

Як зберегти

Power BI Desktop → File → Save As → «Power BI project files (*.pbip)». Desktop створить структуру папок і .gitignore автоматично. Під час Save As ти сам обираєш папку на диску — саме вона стане коренем майбутнього Git-репозиторію.

Стан на 2026. Звітна частина PBIP за замовчуванням використовує формат PBIR (нова модульна структура). .pbix лишається основним контейнером і нікуди не зникає — він теж усередині перейшов на PBIR-метадані.
Скажу прямо: перехід на .pbip — найкраще рішення, яке ти ухвалиш цього року у роботі зі звітами. Роки я тримав .pbix у Git «про всяк випадок», і Git там був марний — суцільний бінарник, де видно лише «щось змінилось», а що саме — ні.

Обери елемент

Натисни на файл або папку зліва.

А так ця сама структура виглядає в реальному Провіднику Windows: .pbip як точка входу, поруч — папка з макетом звіту й папка з моделлю даних (усередині — TMDL-файл на кожну таблицю).

Не сприймай ці папки як окремі файли, які можна комітити вибірково. Уся ця структура — один звіт як цілісна збірка, тому й комітиться вона завжди цілком: git add . на корені проєкту, а не по одному файлу з definition/.
Модуль E — PBIP / PBIR

Формати всередині: PBIR та TMDL

PBIP — це контейнер. Звіт усередині описано форматом PBIR, модель — мовою TMDL.

Ціль урокучитати формати сторінок звіту (PBIR) і моделі (TMDL).
Семантична модель vs звіт
Модель — таблиці, зв'язки, міри (логіка й дані). Звіт — сторінки з візуалами, що показують модель.
Простими словами: модель — це «кухня й рецепти», звіт — «подача страви на тарілці».

TMDL — мова опису моделі

TMDL (Tabular Model Definition Language)
Текстова мова опису моделі з відступами замість дужок. Людиночитна, чудово діфиться.
Простими словами: «модель словами»: рядок за рядком описано кожну таблицю, колонку й міру.
table Sales
    measure 'Total Sales' = SUM(Sales[Amount])
        formatString: #,0
        lineageTag: 3f9a-...-c21

    column Amount
        dataType: double
На реальному проєкті я перевіряв: колега тихцем поміняв формулу міри в model.tmdl, і git diff показав рівно два рядки — було й стало — за секунди. З .pbix та сама ситуація означала б відкривати звіт і вручну звіряти кожну міру: .pbix у Git завжди був чорною скринькою.

PBIR — формат звіту

PBIR (Power BI Enhanced Report Format)
Модульний JSON-опис звіту: сторінки, візуали, закладки — кожне в окремому файлі в definition/.
Простими словами: звіт «по деталях»: кожен візуал — окремий файл, тому Git точно бачить, що саме змінилось.
lineageTag
Унікальний GUID, присвоєний кожному об'єкту моделі. Стабілізує його ідентичність між середовищами.
Простими словами: «незмінний серійний номер» міри чи таблиці. Паралельна робота двох гілок робить його типовим джерелом конфліктів (модуль F).
Інструмент. TMDL зручно редагувати у VS Code (IntelliSense) або Tabular Editor; PBIR JSON — у VS Code з його схемами для перевірки.
Модуль E — PBIP / PBIR

Читаємо diff PBIP: справжні зміни vs шум

Змінив один візуал, натиснув Ctrl+S — а Git показує десятки змінених рядків у кількох файлах. Спокійно: більшість із них — технічний шум. Навчимося за 30 секунд відрізняти одне від одного.

Ціль урокуза 30 секунд відрізняти в diff справжні зміни від технічного шуму й не комітити наосліп.

Чому diff більший за твою правку

Power BI Desktop не редагує файли точково. Під час збереження він переписує файли цілком і дорогою перегенеровує службові поля. Тому поруч зі змістовною правкою в diff зʼявляються рядки, яких ти «не писав».

Шум у diff
Рядки, які змінилися через перегенерацію файлів інструментом, а не через твою свідому правку. Логіки звіту й моделі вони не змінюють.
Простими словами: як пил на склі — заважає дивитися, але картини не змінює. Приклад із Power BI: після збереження в diff «змінився» lineageTag міри, хоча її DAX той самий.

Справжні зміни — те, заради чого й дивимось diff

  • DAX-вираз міри у .tmdl-файлі: SUM → CALCULATE, новий фільтр, інша колонка.
  • Новий стовпець чи звʼязок у TMDL: додався блок column або relationship.
  • Поле чи фільтр візуала у PBIR JSON: в описі візуала зʼявилась інша міра або інша умова фільтра.

Технічний шум — можна не вчитуватися

  • lineageTag — службовий GUID: Desktop подекуди перегенеровує його при збереженні.
  • Координати й розміри візуалів — x, y, z, width: посунув візуал на піксель — Git уже бачить зміну.
  • Порядок властивостей у JSON — той самий зміст, інша послідовність рядків.

Приклад: два фрагменти одного diff

# tables/Sales.tmdl — шум: перегенерований GUID, логіка та сама
- lineageTag: 3f9a-72c1-…-c21
+ lineageTag: 8e4d-19bb-…-b07
# tables/Sales.tmdl — зміст: змінився DAX міри, це і є твоя правка
- measure 'Total Sales' = SUM(Sales[Amount])
+ measure 'Total Sales' = CALCULATE(SUM(Sales[Amount]), USERELATIONSHIP(Dates[Date], Sales[ShipDate]))

Спробуй сам: Desktop щойно перезаписав файли проєкту, і перед комітом ти хочеш глянути зміни конкретного файлу.

А тепер потренуй саме ЧИТАННЯ. Нижче — два реальні PBIP-diff з підсвіткою: зелене — додано, червоне — прибрано. Прочитай кожен і дай відповідь.

Правила на щодень

  • Перед комітом дивись список файлів у Sourcetree (вкладка File Status): чи очікував ти кожен змінений файл.
  • Файл змінився, а ти його «не чіпав» — найпевніше шум перегенерації. Відкрий diff і переконайся, що змістовних правок там немає.
  • Комітити одразу після збереження — менший diff: одна правка = один невеликий читабельний коміт.
Навіщо це вміння. Той самий шум стає джерелом конфліктів, коли дві гілки паралельно перегенеровують ті самі службові поля. Як розвʼязувати такі конфлікти — в уроці «Конфлікти у PBIP-файлах» модуля 07.
Модуль E — PBIP / PBIR

Налаштування PBIP-репозиторію з нуля

Покрокова послідовність: від .pbix до спільного репозиторію на сервері.

Ціль урокупровести проєкт від збереження як PBIP до першого push.

Крок 1 — зберегти як PBIP

Збережи свій звіт як .pbip: у Power BI Desktop виконай File → Save As → «Power BI project files (*.pbip)» і обери папку. Desktop створить структуру й .gitignore.

Ось як виглядає це вікно збереження:

Далі працюєш як звик: відкриваєш .pbip, правиш звіт, тиснеш Ctrl+S — Power BI Desktop перезаписує текстові файли в цій папці.

Крок 2 — увімкнути Git у папці

cd <шлях_до_папки_з_кроку_1>          # перейти в папку проєкту
git init                  # увімкнути контроль версій
git add .                 # усі файли проєкту у staging
git commit -m "Початковий стан PDP"
Де взяти свій шлях? Подивись у заголовку вікна Power BI Desktop або в Провіднику — саме туди Desktop зберіг папку на Кроці 1.
Перевірка перед першим комітом: виконай git status і переконайся, що cache.abf та localSettings.json НЕ потрапили у список (їх має ховати .gitignore). Якщо вони видні — див. модуль F, кейс про git rm --cached.
Подивись очима перед комітом. Перш ніж виконати git commit, відкрий папку проєкту в Sourcetree — на вкладці File Status видно список змінених файлів і diff кожного. Те саме у VS Code: панель Source Control (Ctrl+Shift+G), клік по файлу відкриває порівняння. Так ти комітиш усвідомлено, а не «все, що назбиралося».

Тепер, після git init, Desktop-збереження одразу видно у git status — Git стежить за папкою.

Крок 3 — під'єднати сервер і відправити

Що відбувається: локально → сервер
git remote add origin <url>   # під'єднати сервер (Azure DevOps / GitHub)
git push -u origin main       # відправити перший раз

Спробуй сам: сервер уже створено, лишилося підʼєднати його до локального PBIP-репозиторію.

Крок 4 — колега забирає копію

Що відбувається: git clone
git clone <url>            # колега завантажує проєкт собі
Щоденний цикл після налаштування. Відкриваєш звіт у Power BI Desktop → вносиш зміни → Ctrl+S → переглядаєш diff у Sourcetree або VS Code → git add . і git commit → git push. Якщо при git pull виникне конфлікт — вирішуєш його у merge-редакторі VS Code; типові конфлікти PBIP розібрано в модулі 07.
Модуль E — PBIP / PBIR

Міграція наявних .pbix у PBIP і тонкі звіти

Реальний старт — рідко порожній проєкт. Частіше це папка legacy .pbix-файлів, частина з яких уже опублікована в Power BI Service. Переводимо їх у PBIP, нічого не зламавши.

Ціль урокуперевести наявний .pbix у PBIP, не зламавши опублікований звіт, і розпізнавати тонкі звіти в репозиторії.

Крок за кроком: із .pbix у PBIP

  • Крок 1. Відкрий .pbix у Power BI Desktop.
  • Крок 2. File → Save As → «Power BI project files (*.pbip)» — і обери папку репозиторію.
  • Крок 3. Закоміть і відправ:
git add .
git commit -m "Міграція звіту: .pbix → PBIP"
git push

Опублікований звіт при цьому не ламається: якщо назви елементів (звіту й семантичної моделі) збігаються, то після підключення робочої області до гілки Update from Git оновить наявний звіт, а не створить дубль — посилання користувачів і підписки збережуться. Детальніше про механізм синхронізації — в уроці модуль 07 · Fabric Git integration.

Куди подіти старий .pbix? Прибери його з робочої папки: в історію він однаково не потрапить (*.pbix уже в .gitignore), а лежачи поруч — плутатиме команду, «яка версія справжня». Поклади в архівну папку поза репозиторієм і тримай там, доки не перевіриш, що PBIP-версія в Service працює як слід.

Закріпи послідовність — розстав кроки міграції у правильному порядку.

Тонкі звіти: коли моделі всередині немає

Тонкий звіт (live connection)
Звіт без власної семантичної моделі: він підключений наживо до спільної моделі, опублікованої окремо.
Простими словами: квартира без власної котельні — тепло приходить із міської мережі. Приклад із Power BI: пʼять звітів відділів читають одну спільну модель продажів.

У PBIP такий звіт виглядає незвично: лише папка .Report та файл definition.pbir із посиланням на датасет — папки .SemanticModel немає взагалі. Комітиться й пушиться так само, як звичайний.

Спільна модель + N звітів

  • Модель — окремо. Її зручно вести за стеком 2 (Tabular Editor) — див. урок про стек 2 на цій сторінці.
  • Звіти — окремо. Кожен тонкий звіт — власна PBIP-папка, маленька й легка для diff.
  • Мала команда? Достатньо одного репозиторію з кількома .pbip поруч: модель і звіти в сусідніх папках.
Модуль E — PBIP / PBIR

Обмеження PBIP та коли його не варто застосовувати

Чесний погляд: PBIP чудовий для контролю версій, але має межі.

Ціль урокузнати межі формату і способи їх обходити.

Обмеження

  • Великі моделі. Дуже великі семантичні моделі дають важкі TMDL-діфи; review ускладнюється.
  • Складні merge. Текстовий merge може дати синтаксично валідний, але логічно зламаний файл (дубль lineageTag, биті зв'язки) — потрібна валідація в Desktop / Tabular Editor.
  • Не все у звіті. Окремі специфічні можливості звіту можуть бути представлені неповно; перевіряй після злиття.
  • Бінарні залишки. Зображення та інші asset-и лишаються бінарними — Git їх не діфить (для великих → Git LFS).
  • Дані поза Git. Дані в репозиторії не зберігаються; партиції / incremental refresh живуть у Service і перебудовуються після деплою.

Коли краще НЕ покладатися лише на PBIP

  • Дуже велика модель з інтенсивними паралельними правками одних і тих самих таблиць → розглянути роботу через Tabular Editor (стек 2) з чітким розподілом зон.
  • Команда без дисципліни короткоживучих гілок → конфлікти накопичуються; спершу впорядкувати процес.
Найболючіший кейс, який я бачив: дві людини паралельно перейменували ту саму міру по-різному. Текстовий merge «зжував» обидві правки в один синтаксично коректний, але семантично биту TMDL — Desktop відкрив модель без помилки, а звіт показав пусті картки. Після merge моделі завжди відкривай і перевіряй — не довіряй тільки «зелений merge» у Git.
Висновок. PBIP лишається основою контролю версій для Power BI; обмеження вирішуються дисципліною (малі гілки, валідація після merge, .gitattributes, LFS для бінарників) та, за потреби, стеком Tabular Editor.
Модуль E — PBIP / PBIR

Стек 2: Semantic Model + Tabular Editor

Альтернатива до «PBIP + Power BI Desktop»: модель редагують у Tabular Editor, який дає точніший контроль і автоматичну перевірку якості.

Ціль урокуредагувати модель у Tabular Editor і перевіряти її BPA.
Tabular Editor
Окремий інструмент для редагування семантичної моделі (таблиць, мір, зв'язків) на рівні TMDL, поза Power BI Desktop.
Простими словами: «редактор моделі під капотом» — швидший і точніший для масових змін мір та метаданих, ніж Desktop.
Best Practice Analyzer (BPA)
Вбудований у Tabular Editor аналізатор, що перевіряє модель за набором правил (іменування, описи, формати, продуктивність).
Простими словами: «лінтер моделі» — підсвічує порушення правил ще до коміту чи деплою.

Робочий цикл стеку 2

git pull                          # свіжий стан
# File → Open → Model from Folder → Model/   (відкрити TMDL-папку в Tabular Editor)
# внести зміни: Measures → New Measure → DAX
# Tools → Best Practice Analyzer → виправити зауваження
# Ctrl+S → зміни записуються у .tmdl файли
git add . && git commit -m "feat(model): YTD Sales measure"
git push                          # далі — Git sync публікує в робочу область (опційно: pipeline у Prod)
Стек 1PBIP + Power BI Desktop. Модель і звіт разом; зручно для звітної частини й загальної розробки.
Стек 2Semantic Model + Tabular Editor. Точний контроль моделі, BPA, масові правки мір; звіт — окремо.

Модуль E — PBIP / PBIR

Що комітити, а що ігнорувати: .gitignore для PBIP

Power BI Desktop кладе у папку проєкту не лише «креслення» звіту й моделі, а й локальний кеш даних. В історію Git має потрапляти тільки перше.

Ціль урокучітко знати, які файли PBIP належать історії, а які — у .gitignore.
.gitignore
Текстовий файл у корені репозиторію зі списком шаблонів шляхів, які Git не відстежує й не пропонує комітити.
Аналогія: список «не пакувати» під час переїзду: сміття й особисті речі сусіда у вантажівку не їдуть.
Комітити.pbip, увесь вміст definition/ (TMDL і PBIR), .gitignore, .gitattributes — усе, що описує звіт і модель текстом.
Ігнорувати**/.pbi/cache.abf — локальний кеш даних (може важити сотні мегабайтів); **/.pbi/localSettings.json — особисті налаштування твого Desktop; *.pbix — бінарні копії поруч із проєктом.

Готовий .gitignore для PBIP-репозиторію

# створюєш файл .gitignore у корені репозиторію (VS Code: File → New File)
# Power BI Desktop сам пропонує такий вміст під час збереження у PBIP:
**/.pbi/localSettings.json
**/.pbi/cache.abf
*.pbix

Практична перевірка: відкрий проєкт у Power BI Desktop, збережи й глянь git status у Git Bash (або список змін у Sourcetree). Якщо серед змінених файлів немає cache.abf — .gitignore працює.

Кеш уже закомічено? .gitignore не діє на файли, які Git уже відстежує. Лікується через git rm --cached — покроково у повному уроці про .gitignore в модулі 07, там само — типові кейси команди.
