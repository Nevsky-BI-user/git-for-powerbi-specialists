Гілка для фічі — не чіпаючи main

Одна фіча — одна гілка: main лишається чистою і завжди готовою до публікації.

Цільстворити фіче-гілку однією командою і закомітити роботу саме в ній

Коротке нагадування. У команді Power BI гілка main — це «вітрина»: саме з неї Git sync (Update from Git) публікує звіт у робочу область. Тому в main потрапляють лише готові зміни, а вся жива робота — нові KPI-картки, експерименти з візуалами — живе у гілках feature/*. Потік простий: feature-гілка → merge у main → push → Git sync (докладніше — у модулі 07).

Гілка в Git — це не копія проєкту, а легка «закладка» на коміті. Команда git switch -c feature/kpi-cards робить одразу два кроки: створює гілку і перемикає тебе на неї. Далі коміт іде вже в неї, а main стоїть на місці незайманою.

Незакомічені зміни не зникають при switch -c — вони «переїжджають» разом з тобою в нову гілку. Тож навіть якщо ти почав правити файли до створення гілки, ще не пізно все зробити правильно.

Ситуація в тренажері: ти вже підправив visual.json для нових KPI-карток, але досі стоїш на main. Створи гілку feature/kpi-cards і закоміть роботу саме в ній — main не має отримати жодного нового коміту.

Два види злиття: fast-forward і merge-коміт

Команда одна — git merge, а результати бувають два: тихе «перемотування» або новий коміт із двома батьками.

Цільна власних руках відчути різницю: коли merge проходить як fast-forward, а коли створює merge-коміт

Нагадування з теорії. Якщо з моменту відгалуження main не зробила жодного власного коміту, то зливати насправді нічого: усі коміти гілки і так лежать «попереду» main. Git просто пересуває мітку main уперед — це і є fast-forward. Нового коміту не зʼявляється, історія лишається однією прямою лінією.

Перший тренажер — саме такий випадок: гілка feature/date-table пішла вперед, а main стояла на місці. Виконай злиття і придивись до схеми: жодного нового коміту — main просто «наздоганяє» гілку.

А тепер — та сама команда, але інша ситуація: поки ти працював у своїй гілці, у main зʼявився власний коміт (колега вже влив свою роботу). Історії розійшлися, і «перемотати» мітку вже неможливо — Git створить спеціальний merge-коміт, який зшиває дві лінії докупи.

Виконай злиття у другому тренажері й подивись на граф: зʼявиться коміт M1 із двома батьками — по одному з кожної гілки.

Схема: коли можливий fast-forward

Перш ніж тиснути Enter після git merge, корисно вміти передбачити, що саме станеться.

Цільпо схемі гілок передбачати, яким буде злиття — fast-forward чи merge-коміт

Правило одне-єдине: fast-forward можливий лише тоді, коли гілка-приймач (зазвичай main) не має власних комітів після точки розгалуження. Є хоч один — Git змушений створити merge-коміт. Прокрути три кроки нижче і подивись на обидві ситуації в русі.

Fast-forward чи merge-коміт?

Закріпи правило міні-перевіркою — без підглядання у схему.

Конфлікт злиття і порятунок із detached HEAD

Конфлікт — це не аварія, а запитання від Git; а detached HEAD — не пастка, якщо знати дві команди.

Цільрозвʼязати merge-конфлікт руками (edit → add → commit) і врятувати коміт із відірваного HEAD

Конфлікт злиття виникає, коли дві гілки змінили ті самі рядки одного файлу — наприклад, ту саму міру в definition/model.tmdl. Git не вгадує, чия версія правильна: він зупиняється і питає тебе. Відповідь завжди складається з трьох кроків: відредагувати файл (прибрати маркери конфлікту і залишити правильний текст), git add, git commit — і злиття завершено.

Слово CONFLICT у терміналі — це питання, а не помилка. Нічого не зламалось і не втратилось: обидві версії рядків лежать у файлі, Git чекає на твоє рішення.

Тренажер: ти зливаєш feature/new-calc у main, і Git зупиняється з CONFLICT у model.tmdl. Заверши злиття руками — edit → add → commit — і на графі зʼявиться коміт злиття M1.

Друга ситуація — detached HEAD («відірвана голова»): це стан, коли ти стоїш не на гілці, а прямо на конкретному коміті, наприклад після git checkout на старий хеш. Коміт, зроблений у такому стані, не належить жодній гілці — і якщо просто перемкнутися геть, він може загубитися (як його шукати потім — розповідає модуль 04).

У тренажері цінний коміт уже зроблено у відірваному стані. Постав на нього гілку rescue командою git branch rescue — і спокійно повертайся на main.

Екзамен: гілки і злиття

Три питання на читання графа історії — без підказок і без тренажера під рукою.

Цільчитати git log --graph і передбачати поведінку гілок після злиття

Тут перевіряється те, що ти щойно робив руками: як виглядає merge-коміт на графі git log --graph і хто його батьки, що стається з фіче-гілкою після злиття, і чим git switch -c відрізняється від git branch. Читай схеми уважно — відповіді у них.
