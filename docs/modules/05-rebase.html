Модуль D — rebase

Що таке rebase

Rebase бере набір комітів і відтворює (replay) їх на новій основі, роблячи історію лінійною. На відміну від merge, що зберігає розгалуження й додає merge-коміт.

Ціль урокурозуміти, що робить rebase і чим він відрізняється від merge.
Саме rebase у цьому відео окремо не розглядається — це тема нашого курсу. Фрагмент вище показує суміжне: вирішення конфліктів злиття (на прикладі GitHub).
rebase (перебазування)
Перенесення комітів з однієї основи на іншу. Git створює нові коміти з тим самим вмістом, але іншим батьком.
Простими словами: «переписати свою гілку так, ніби ти починав її від свіжого стану main».
replay (відтворення)
Повторне накладання змін коміту поверх нової основи зі створенням нового коміту.
Простими словами: Git ніби «повторно набирає» твої зміни на новому місці.

Візуалізація: rebase переносить коміти

До та після rebase

Ключовий механізм

Rebase не пересуває старі коміти — він створює нові з тим самим вмістом, але іншим батьком, отже з новими SHA. Старі лишаються orphaned і згодом зникають.

git rebase main          # перенести гілку поверх main
git rebase -i HEAD~4      # інтерактивно: останні 4 коміти
git rebase --continue     # після вирішення конфлікту
git rebase --abort        # скасувати rebase
Де запускати. Rebase найпростіше робити в терміналі Git Bash — Git на кожному кроці пише підказку, що робити далі. У Sourcetree те саме — правий клік по гілці main → «Rebase current changes onto main», а якщо щось пішло не так, кнопка Abort робить те саме, що git rebase --abort.
Золоте правило rebase. Не роби rebase комітів, які вже на сервері й спільні. Rebase — для локальних, ще не опублікованих комітів.

Модуль D — rebase

Конфлікти злиття

Конфлікт виникає, коли дві гілки змінили ту саму ділянку файлу по-різному. Git не може вирішити сам і передає вибір тобі.

Ціль урокуспокійно вирішувати конфлікт: маркери, вибір, add, commit.
conflict (конфлікт злиття)
Ситуація, коли Git не може автоматично об'єднати два набори змін, бо вони чіпають однакові рядки.
Простими словами: двоє написали різний текст в одному рядку — Git питає, чий лишити.

Як виглядає у файлі

<<<<<<< HEAD
measure 'Total Sales' = SUM(Sales[Amount])
=======
measure 'Total Sales' = SUMX(Sales, Sales[Qty] * Sales[Price])
>>>>>>> feature/new-calc
  • Між <<< і === — версія поточної гілки (HEAD).
  • Між === і >>> — версія гілки, яку зливаєш.
  • Прибрати маркери, лишити правильний варіант (або поєднати вручну).

Процес

git status              # файли з конфліктами
# … редагуєш, прибираєш маркери …
git add model.tmdl      # позначити вирішеним
git commit              # merge; або:
git rebase --continue   # rebase
Робочий сценарій з інструментами. Робиш git pull у Git Bash (або тиснеш Pull у Sourcetree) — і бачиш CONFLICT (content): Merge conflict in model.tmdl. Відкриваєш файл у VS Code: він сам підсвічує маркери й пропонує merge-редактор із кнопками Accept Current / Accept Incoming — одним кліком обираєш свою чи чужу версію (або поєднуєш вручну). Далі стандартно: git add, git commit, git push. Усі ці інструменти ти ставив у модулі 00 — Старт та інструменти.
Під час rebase конфлікт спрацьовує покомітно. Переносиш 4 коміти — конфлікт може виникнути до 4 разів. Sourcetree: Resolve Conflicts → Continue Rebase.
