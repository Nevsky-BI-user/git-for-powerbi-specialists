Старт

Вітаю в курсі Git для Power BI

Якщо ти ще жодного разу не відкривав термінал і не знаєш, що таке «коміт» — це нормально. Курс веде з нуля, простою мовою, з прикладами саме під Power BI.

Ціль урокупобачити, чому без контролю версій важко з PBI-звітами, познайомитися з інструментом, який це розвʼязує, і зрозуміти, як користуватись матеріалом курсу.

Знайомі ситуації?

Перш ніж говорити про інструменти — впізнай себе. Було таке?

Зламалось перед самою демонстрацією

Поміняв міру за пів години до зустрічі — і звіт розсипався. А яким усе було вчора, коли ще працювало? Того файлу вже нема.

Тека з пʼятьма «фінальними» версіями

Report_v3_final.pbix, Report_v3_final_FINAL.pbix, Report_v3_final_FINAL_ostatochno.pbix — і вже ніхто, включно з тобою, не памʼятає, чим вони відрізняються.

Страшно чіпати робочий звіт

Усе працює — і саме тому лячно щось міняти. Зламаєш — і шляху назад немає, лишається тільки переробляти з нуля.

Файл зник разом із ноутбуком

Ноутбук завис, диск помер, хтось перезаписав файл на спільному диску — і тижні роботи зникають за секунду.

Впізнав хоча б одну ситуацію? Це не поодинокий випадок — так у Power BI-командах губляться тижні роботи буквально щотижня. І річ не в недбалості: руками таке контролювати неможливо.

Найчастіша реакція на цей біль — уже знайома тека з попереднього списку. Хочеш зберегти проміжний стан — копіюєш файл і дописуєш у назву щось на кшталт _v2 чи _final. Це і є спроба контролю версій руками, тільки без жодного інструменту: кожна копія — це «версія», а назва файлу — єдине, що про неї розповідає. Працює, поки версій дві-три. А далі: копії пухнуть і займають дедалі більше місця; назви брешуть — «final» рідко буває останнім; всередину не заглянеш, не порівнявши файли вручну візуал за візуалом; а якщо в пʼятій версії щось зламалось і хочеш повернути лише одну конкретну зміну — доведеться відкочуватись до всієї версії цілком, разом з усім іншим, що встиг зробити після.

Що таке Git (простими словами)
Git — програма, яка робить те саме, що ти щойно робив копіями файлів, тільки по-розумному: сама веде історію версій усієї папки проєкту. Кожне збереження (коміт) — це «знімок» стану, який нікуди не зникає. У будь-який момент можна повернутись до будь-якого знімка, порівняти два між собою або вести кілька паралельних чернеток одночасно — без жодної ручної копії.
Аналогія: машина часу для папки проєкту — тільки запис у ній робиться сам, щоразу коли тиснеш «зберегти версію». Одне чесне речення на довіру: це безкоштовний стандарт, яким уже понад 20 років користується весь світ розробки, — і він так само добре працює для теки з PBI-звітом, як і для коду.

Тепер, коли зрозуміло, що дає Git, — повернімось до самого файлу звіту. Ти будуєш звіт у Power BI Desktop, тиснеш «Зберегти» — і отримуєш звичний файл .pbix. Це нормальна відправна точка: так зберігає кожен Power BI-розробник. Проблема в тому, що .pbix — це один суцільний архів: усередині одразу і модель даних, і DAX-міри, і макет сторінок, і кеш. Зміниш один рядок формули чи пересунеш одну візуалізацію — і для Git (та будь-якої іншої програми, що порівнює версії) файл виглядає «весь інакший»: жодного знімка, де видно саме твою правку.

Курс проведе тебе одним ланцюжком кроків, який перетворює цю коробку на щось, з чим уміє працювати Git. Ось увесь шлях одразу — це головна мапа курсу, і кожен наступний модуль розкриває один її відрізок:

Коротко: у Power BI Desktop зберігаєш проєкт форматом .pbip замість .pbix — і замість однієї коробки отримуєш папку зі звичайними текстовими файлами (модуль 06). Ці файли Git уміє відстежувати рядок у рядок: git init і перший коміт фіксують стан проєкту (модулі 01–02), push відправляє його на сервер — GitHub чи Azure DevOps (модуль 03), а Git sync у робочій області Power BI Service забирає звідти готовий звіт (модуль 07). Той самий ланцюжок веде від файлу на твоєму диску до звіту, який бачить бізнес.

До кожного «загального» модуля курсу додано фрагмент безплатного відеокурсу українською від Нікіти Тимошенка — з таймкодом саме на цю тему. Відео вбудовано офіційним плеєром YouTube (з дозволу автора) і вантажиться лише після кліку. Модулі про PBIP (06) і просунуті техніки (08) відео не мають — такого контенту українською поки немає, тож там спираймося на текст курсу.

Навіщо Git, якщо я працюю сам

Усі чотири ситуації з початку уроку — це саме ти, сам на сам зі своїм звітом, без жодного колеги поруч. У реальній роботі з Power BI над одним звітом рідко одночасно сидять двоє розробників, і Git на перший погляд виглядає як інструмент «для команди», який тобі ніби не потрібен. Але «я один» не забирає жодної з причин, чому він корисний. Навпаки: без колеги, який щось памʼятає замість тебе, історія змін — єдине, що памʼятає за тебе. Ось що саме з переліку на початку уроку вирішує звичайний, самостійний Git:

  • Історія власних змін. Навіть якщо на питання «хто змінив» завжди одна відповідь — ти, лишаються «коли» і «що саме». Через два місяці відкриєш лог комітів і побачиш, що міру Sales YTD востаннє чіпав 14 березня, а не «десь навесні, здається».
  • Більше не страшно чіпати робочий звіт. Той страх зі сценарію на початку уроку — «а раптом зламаю і не поверну» — Git знімає: створюєш гілку, пробуєш інший підхід до розрахунку маржі; не вийшло — просто видаляєш гілку, а основний звіт навіть не помітив спроби; вийшло — вливаєш назад.
  • «Ще вчора звіт працював». Точно та ситуація з демонстрацією вище: поміняв щось перед зустріччю — і не памʼятаєш, яким усе було до цього. З Git це не вечір розслідування, а одна команда: відкрити вчорашню версію і побачити, чим вона відрізняється від сьогоднішньої.
  • Пошук моменту, коли зламалась міра. Звіт раптом показує не ті числа, а коли саме це почалось — незрозуміло. Історія комітів дозволяє пройти назад крок за кроком (а пізніше, у модулі 08, — навіть автоматично, командою git bisect) і знайти точний коміт, де щось змінилося.
  • Кінець теці з Report_v3_final_FINAL_ostatochno. Памʼятаєш плутанину зі сценарію вище? Версії живуть не в назві файлу, а в історії комітів: одне чисте імʼя файлу назавжди, всі версії — всередині Git.
  • Ноутбук більше не єдина копія. Запушив у GitHub або Azure DevOps — і той жах зі сценарію вище («файл перезаписали, тижні роботи зникли») більше не страшний: копія проєкту живе на сервері, незалежно від того, що станеться з твоїм диском.

Спільна робота з колегами — це бонус, який Git дає, коли він знадобиться, а не передумова, без якої соло-розробнику начебто нема сенсу починати.

Що ти вмітимеш наприкінці

  • Впевнено працювати в терміналі: створювати папки й файли, рухатись між ними.
  • Зберігати версії проєкту (коміти), повертатись назад і не боятися щось зламати.
  • Працювати в команді: гілки, злиття, вирішення конфліктів.
  • Вести Power BI-проєкт у форматі PBIP під Git і публікувати його через Git sync робочої області (за потреби — deployment pipeline у Prod).

Як користуватись курсом

  • Меню ліворуч — модулі від простого до складного. Його можна сховати кнопкою згортання.
  • Пошук угорі ліворуч шукає по всьому курсу.
  • Схеми з кнопками «Назад / Далі» показують процес покроково — гортай їх.
  • Міні-вправи під темами — одразу перевір себе одним кліком. Наприкінці кожного модуля є більші вправи.
  • Далі в цьому модулі — знайомство з інструментами і розгортання робочого місця: Git, Git Bash, GitHub / Azure DevOps, VS Code, Sourcetree.
Порада: не пропускай модуль «Термінал і файли», навіть якщо здається елементарним. Усі команди Git вводяться там само, і далі весь курс спирається на ці базові навички.
Старт та інструменти

Карта інструментів: хто за що відповідає

«Git» у розмовах — це часто ціла екосистема з пʼяти-шести програм. Розкладемо її по поличках, щоб далі не плутатись.

Ціль урокурозрізняти двигун (Git), сховище (GitHub / Azure DevOps) і клієнти (Git Bash, VS Code, Sourcetree) — і розуміти, де в цій картині Power BI.
Git (двигун)
Програма без вікон, яка веде історію версій у прихованій папці .git. Усе інше — лише способи натискати на неї.
Аналогія: двигун автомобіля. Сам по собі невидимий, але саме він везе.
Хостинг репозиторіїв (GitHub, Azure DevOps Repos)
Сервер, де лежить спільна копія репозиторію, плюс вебінтерфейс: перегляд історії, налаштування доступів, автоматизації.
Аналогія: хмарний диск для проєкту + офіс, де команда домовляється про правила.

Ось як виглядає репозиторій на GitHub у браузері:

Клієнт Git (Git Bash, VS Code, Sourcetree)
Програма, через яку ти віддаєш команди двигуну: текстом у терміналі або кнопками у графічному інтерфейсі.
Аналогія: кермо і педалі. Обираєш, що зручніше саме тобі, — їде однаково той самий двигун.
Типова помилка на старті — казати «запушив у Git», маючи на увазі GitHub. Насправді Git у тебе вже працює локально в ту секунду, коли зроблено перший коміт; GitHub чи Azure DevOps потрібні лише тоді, коли хочеш поділитися результатом з командою. Ці дві речі легко переплутати на старті — і це нормально.

Хто є хто в екосистемі

  • Power BI Desktop — тут ти редагуєш звіт і модель; у форматі PBIP він зберігає проєкт текстовими файлами, які розуміє Git.
  • Git — рушій версій: коміти, гілки, історія. Працює локально на твоєму компʼютері.
  • Git Bash — термінал для Windows, встановлюється разом із Git. Саме тут вводяться команди з цього курсу.
  • GitHub — найпопулярніший хостинг: особисті проєкти, портфоліо, відкритий код.
  • Azure DevOps Repos — корпоративний хостинг від Microsoft: єдиний вхід компанії, політики гілок, pipelines. Типовий вибір там, де вже є Power BI і Fabric.
  • VS Code — безплатний редактор коду: перегляд diff, правка TMDL/JSON, вирішення конфліктів.
  • Sourcetree — графічний Git-клієнт: історія у вигляді графа, commit і push кнопками.
  • Tabular Editor — просунута робота з семантичною моделлю (детально в модулі 06).
  • Fabric Git integration — міст між репозиторієм і workspace Power BI (детально в модулі 07).
Порада: не намагайся опанувати все одразу. Мінімальний робочий набір: Power BI Desktop + Git + один зручний клієнт (Sourcetree або VS Code). Решта додасться сама.
Старт та інструменти

Встановлюємо Git і Git Bash

Перший крок розгортання інфраструктури — поставити двигун. Разом із ним безплатно приїде і термінал.

Ціль урокувстановити Git на Windows, перевірити встановлення і представитись системі.

Кроки встановлення

  • Завантаж інсталятор з офіційного сайту git-scm.com.
  • Запусти і тисни Next: стандартні налаштування підходять. Пункт про кінці рядків залиш як є (значення за замовчуванням підходить) — що таке checkout і commit, розберемо в модулях 02–03.
  • Після встановлення у меню «Пуск» зʼявиться Git Bash — термінал, у якому працюватимеш весь курс.
Відкрий git-scm.com, натисни Download for Windows, запусти завантажений інсталятор і клацай Next до кінця — типові налаштування підходять.
Git Bash
Термінал для Windows з Unix-командами (pwd, ls, cd), який іде в комплекті з Git.
Простими словами: вікно, куди ти друкуєш команди. На Windows воно дає ті самі команди, що бачать колеги на macOS чи Linux, — тому всі інструкції в інтернеті працюють однаково.

Перевірка і перше налаштування

Відкрий Git Bash (Пуск → напиши "Git Bash" → Enter) і введи по черзі:

bash
# чи встановився Git: має вивести номер версії $ git --version # представся: це імʼя та email підписуватимуть кожен твій коміт $ git config --global user.name "Ім'я Прізвище" $ git config --global user.email "you@company.com"
Спокійно: тут неможливо нічого зламати. Команда git config --global лише підписує майбутні коміти твоїм імʼям і поки що не чіпає жодного репозиторію — це налаштування можна змінити коли завгодно, хоч через рік.
Порада: у корпоративному ноутбуці Git може бути вже встановлений або ставиться через ІТ-відділ. Просто відкрий Git Bash і перевір git --version — якщо версія вивелась, крок виконано.
Старт та інструменти

GitHub і Azure DevOps Repos: дім для репозиторію

Git на твоєму компʼютері вже вміє все. Але щоб працювати командою і не втратити роботу разом із ноутбуком, потрібна спільна копія на сервері.

Ціль урокузрозуміти, що таке хостинг репозиторіїв, чим GitHub відрізняється від Azure DevOps, і як отримати репозиторій собі на компʼютер.
Віддалений репозиторій (remote)
Копія репозиторію на сервері хостингу. Ти надсилаєш туди коміти (push) і забираєш чужі (pull).
Аналогія: спільна шафа з документами в офісі. Кожен працює за своїм столом (локально), а результат складають у шафу, звідки його бачать усі.

Який хостинг обрати

  • GitHub — для особистих проєктів, навчання, портфоліо. Реєструєшся, тиснеш New repository, обираєш private або public — репозиторій готовий за хвилину.
  • Azure DevOps Repos — стандарт у компаніях з Microsoft-стеком: вхід через робочий обліковий запис, політики гілок, pipelines для автоматизації і пряма інтеграція з Fabric (Git sync workspace). Репозиторій створюється в межах проєкту: Project → Repos → New repository.

Для курсу підійде будь-який: команди Git однакові. Далі в прикладах здебільшого Azure DevOps — бо саме він найчастіше зустрічається в корпоративній роботі з Power BI.

Забираємо репозиторій собі

Команда git clone копіює на твій комп'ютер увесь репозиторій із сервера — файли, всю історію комітів і зв'язок із сервером (origin) вона підключає сама. Детальна схема того, що саме тоді відбувається, — в уроці «Локальний і хмарний репозиторій».

# адресу дає кнопка Clone на сайті хостингу
git clone https://dev.azure.com/org/project/_git/pbi-reports
Тепер уяви: новий аналітик приєднався до команди в четвер. Один git clone — і в нього вже є доступ до кожної версії моделі за останні два роки, включно з тими правками, які ти сам давно забув.

Під час першого push або clone хостинг попросить увійти: зазвичай відкриється вікно браузера (за це відповідає Git Credential Manager). Про SSH-ключі та Personal Access Token — окремий урок у модулі 03.

Старт та інструменти

VS Code: редактор із вбудованим Git

PBIP — це текстові файли. А для роботи з текстом потрібен хороший редактор.

Ціль урокувикористовувати VS Code для перегляду змін у PBIP-файлах і вирішення конфліктів.
VS Code (Visual Studio Code)
Безплатний редактор коду від Microsoft із вбудованою підтримкою Git: панель Source Control, перегляд diff, merge-редактор для конфліктів.
Простими словами: «блокнот на стероїдах». Відкриваєш папку PBIP-проєкту — і бачиш, що саме змінилось у моделі, рядок у рядок.

Що ти робитимеш у VS Code найчастіше

  • Дивитись зміни: панель Source Control (Ctrl+Shift+G) показує змінені файли; клік по файлу — і бачиш diff: що було ліворуч, що стало праворуч.
  • Правити TMDL/JSON: дрібну правку міри можна зробити прямо в measure.tmdl, не відкриваючи Power BI Desktop.
  • Вирішувати конфлікти: merge-редактор пропонує кнопки Accept Current / Accept Incoming / Accept Both — детально в модулі 05.
  • Комітити: кнопка Commit у тій самій панелі — якщо не хочеться перемикатись у термінал.

Навіщо взагалі відкривати diff, а не одразу комітити? Тому що PBIP-файли міняються й тоді, коли ти нічого не чіпав навмисно: перетягнув візуал на пів пікселя, Desktop перерахував внутрішній ідентифікатор об'єкта, або хтось зберігся з іншим масштабом сторінки. У diff таку зміну видно одразу — підсвічений рядок у файлі макета, хоча зовні звіт виглядає так само. Побачив зайве — розберешся, перш ніж воно поїде в коміт разом із потрібною правкою.

Корисні розширення

  • TMDL — підсвітка синтаксису TMDL-файлів семантичної моделі.
  • GitLens — показує біля кожного рядка, хто і коли його змінив.
Не плутай: VS Code і Visual Studio — різні продукти. Для роботи з PBIP потрібен саме легкий VS Code.
Старт та інструменти

Sourcetree та інші графічні клієнти

Команди — це точно і швидко. Але дивитись історію та збирати коміт по файлах часто зручніше мишкою.

Ціль урокувиконувати щоденний цикл (pull → зміни → commit → push) у графічному клієнті й розуміти, коли GUI зручніший за термінал.
Sourcetree
Безплатний графічний Git-клієнт від Atlassian: історія комітів у вигляді графа, список змінених файлів, кнопки Stage / Commit / Push / Pull.
Аналогія: автоматична коробка передач. Усередині той самий Git, але керування — двома кнопками. Для щоденної рутини цього достатньо.

Типовий цикл у Sourcetree

  • Зранку — кнопка Pull: забрав свіжі зміни колег.
  • Попрацював у Power BI Desktop, зберіг PBIP — у Sourcetree одразу видно список змінених файлів і diff кожного.
  • Натиснув Stage All — увесь PBIP-звіт у staging одним рухом, без відбору окремих файлів (файли моделі й макета взаємоповʼязані, тому комітиш цілісно), написав повідомлення, натиснув Commit.
  • Push — зміни поїхали на сервер.

Альтернативи

  • GitHub Desktop — ще простіший, ідеальний у парі з GitHub.
  • Fork — швидкий і охайний клієнт для Windows/macOS.
  • Вбудований Source Control у VS Code — якщо не хочеш ставити окрему програму.
Практичне правило: рутина (подивитись зміни, закомітити, запушити) — у GUI; точні або рідкісні операції (rebase, reflog, bisect) — у терміналі. Курс навчить обох способів.
Старт та інструменти

Уся зв'язка разом: від звіту до продакшену

Тепер збери всю картину: як одна правка міри проходить крізь усі інструменти — від Power BI Desktop до звіту, який бачить бізнес.

Ціль урокубачити повний шлях зміни: Power BI Desktop → Git → Azure DevOps → робоча область Power BI — і знати, який інструмент відповідає за кожен крок.

Один робочий день — покроково

  • 1. Power BI Desktop. Відкриваєш PBIP-проєкт, виправляєш міру Sales YTD, зберігаєш. Desktop переписує текстові файли у папці проєкту.
  • 2. Sourcetree (або VS Code). Бачиш, які саме файли змінились, переглядаєш diff: у measure.tmdl змінився один рядок DAX.
  • 3. Коміт. git add . — весь звіт у staging: PBIP-проєкт це цілісна збірка (файли моделі й макета взаємоповʼязані), тому комітиш його завжди цілком, а не файл за файлом → повідомлення fix: correct YTD filter → Commit. Кнопкою в Sourcetree чи командою в Git Bash — результат однаковий.
  • 4. Push в Azure DevOps. Зміна тепер на сервері, її бачить уся команда.
  • 5. Git sync. Робоча область у Power BI Service підтягує зміну з гілки (Update from Git) — звіт опубліковано, бізнес бачить виправлення. Для звичайної публікації це фініш.
  • 6. (За потреби) Deployment pipeline. Якщо команда тримає окремі області Dev і Prod, додається ще один крок: після перевірки в Dev натискаєш Deploy — pipeline промоутить вміст у Prod. Це опційний рівень, а не обовʼязкова частина процесу.

А якщо на кроці 4 сервер відповів, що колега вже змінив той самий файл? Тоді зʼявляється ще один інструмент: VS Code відкриває конфліктний файл, ти обираєш правильну версію в merge-редакторі — і повторюєш push. Цьому присвячений модуль 05.

Запамʼятай цей ланцюжок — він повторюватиметься в кожному модулі курсу, щоразу глибше: команди — у модулях 01–03, виправлення помилок — у 04–05, PBIP — у 06, командна робота і деплой — у 07.
