Push: відправити роботу на сервер

Коміт без push — це лист, який ти написав, але не надіслав: команда його не бачить.

Цільпушити коміти на сервер і публікувати нову гілку з першим push -u
Перший push на сервер завжди трохи хвилює — ніби надсилаєш лист і не бачиш, чи дійшов. Я поруч: застрягнеш — тисни Підказку, це не соромно; соромно — не спробувати самому.

Сервер (Azure DevOps або GitHub) — це місце зустрічі команди: спільна копія репозиторію, до якої всі пушать і з якої всі забирають зміни. Саме звідти Fabric через Git sync (Update from Git) підтягує оновлення у робочу область. Тому базовий потік такий: робота у feature/* → merge у main → git push → Git sync публікує зміни.

Поки ти не виконав git push, усі твої коміти живуть лише на твоєму диску. Модель може бути готова, міри пораховані — але для колег і для Fabric нічого не змінилось. git status завжди чесно скаже, наскільки ти попереду сервера: ahead of origin/main by N commits.

Тренажер лише симулює сервер — push тут нікуди реально не йде, тож пробуй сміливо. Для справжньої практики створи окремий тестовий репозиторій на GitHub (порожній, не той, де живе робочий звіт), підключи його командою git remote add origin і повтори push у Git Bash — лише так побачиш свіжий коміт на сторінці репозиторію в браузері.

Перша задача: ти зробив локально два коміти зі змінами звіту, а сервер досі бачить лише перший. Подивись git status, потім відправ роботу — на схемі бейдж «сервер» наздожене твою гілку.

Друга ситуація — нова гілка. Ти створив feature/refresh-fix локально, але сервер про неї ще нічого не знає, тому простий git push відмовить. Не лякайся: Git у тексті відмови сам підкаже повну команду з -u origin. Спробуй.

Прапорець -u (upstream) — це «запамʼятай звʼязок»: Git привʼяже локальну гілку до серверної, і всі наступні рази вистачить короткого git push.

Fetch і pull: спершу подивитися, потім забрати

Fetch — розвідка без наслідків, pull — розвідка плюс застосування. Звичка розрізняти їх рятує від сюрпризів.

Цільрозрізняти fetch (безпечна розвідка) і pull (застосувати зміни) та читати стан origin/main

git fetch лише оновлює твоє знання про сервер: мітка origin/main пересувається туди, де реально стоїть сервер, а твої локальні файли й гілки не змінюються ані на байт. Тому fetch безпечний завжди. git pull — це fetch плюс злиття: він і дізнається нове, і застосовує його до твоєї гілки. Якщо власних нових комітів у тебе немає, злиття вироджується у простий fast-forward — гілка просто «доїжджає» вперед.

Проклацай схему: перший кадр — стан одразу після fetch, другий — що робить pull. Деталі теорії — у модулі 03.

Що бачить fetch, що робить pull

Тепер руками. Колега запушив коміт із новою таблицею, поки ти працював. Виконай git fetch — побачиш на схемі, що origin/main пішов уперед, а файли ще не змінені. Потім забери зміни собі.

Відхилений push: класика командної роботи

«! [rejected] non-fast-forward» — не аварія, а ввічливе «спершу забери чуже, потім віддавай своє».

Цільспокійно реагувати на «! [rejected] non-fast-forward»: pull --rebase і повторний push

Це найчастіша «страшна» відмова Git у командній роботі — побачиш її щотижня. Значить вона просте: на сервері є коміт, якого немає в тебе. Колега встиг запушити раніше, і якби сервер прийняв твій push, робота колеги затерлася б. Відмова — це захист, а не помилка.

Розвʼязка у два кроки: git pull --rebase забирає коміт колеги і перегра́є твій коміт поверх нього (C2 стане C2′ — той самий зміст, нове місце в історії), а повторний git push уже проходить. Історія залишається лінійною, без зайвого merge-коміту. Як команди домовляються про такі правила — у модулі 07.

Відтвори сценарій у тренажері: спробуй git push, отримай відмову, виправ ситуацію так, щоб історія лишилась лінійною, і допуш свою міру.

А тепер — «бос» цієї теми: та сама відмова push, але цього разу ви з колегою змінили один і той самий файл. Реплей git pull --rebase зупиниться конфліктом could not apply — і завершується він не комітом, а командою git rebase --continue (відступити можна через git rebase --abort).

Ніколи не «пробивай» відмову через git push --force на спільній гілці: сервер прийме, але коміт колеги буде затерто. Force-push — інструмент для своїх особистих гілок, не для main.

Екзамен: синхронізація з сервером

Три ситуації з реального робочого дня: відмова push, стан після fetch і вибір між clone та pull.

Цільправильно реагувати на відмови push, розуміти безпечність fetch і різницю clone/pull

Тут без тренажера: читаєш вивід терміналу — обираєш правильну реакцію, як у житті. Якщо якесь питання змусило вагатися, повернись до відповідної секції вище й переграй тренажер ще раз.
