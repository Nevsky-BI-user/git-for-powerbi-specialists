Читаємо PBIP-diff і конфлікти

Diff PBIP-проєкту завжди «шумить» — екзамен починається з уміння бачити крізь цей шум.

Цільвідрізняти змістовні зміни TMDL від технічного шуму і правильно розвʼязувати конфліктні маркери

Коли Power BI зберігає проєкт у форматі PBIP, у файлах на кшталт definition/model.tmdl поруч зі змістовними речами (формули мір, звʼязки, назви) живуть технічні рядки: lineageTag, ordinal, службові ідентифікатори. Power BI переписує їх сам, і вони регулярно зʼявляються у diff, хоча по суті нічого не змінюють. Хороший коміт-меседж описує змістовну зміну — нову умову у формулі DAX, перейменовану міру, доданий звʼязок — а не перший-ліпший рядок, який трапився у виводі git diff.

Друга навичка — конфліктні маркери. Коли дві гілки змінили один рядок міри, Git вставляє у файл підказки <<<<<<<, ======= і >>>>>>>. Розвʼязати конфлікт означає лишити одну робочу версію формули і прибрати всі маркери — файл TMDL з маркерами всередині просто зламає модель. Якщо треба освіжити теорію, зазирни в модуль 06.

І третє — прогноз злиття по git log: діапазон main..feature показує, що є у feature і чого немає в main; зворотний діапазон — навпаки. Пара цих виводів дозволяє ще до команди git merge сказати, буде fast-forward чи merge-коміт.

Порада. Читаючи PBIP-diff, спершу подумки викресли рядки з lineageTag та іншим технічним шумом — а вже потім питай себе: «що тут змінилося насправді?».
Екзамен тут — без ризику: тренажер перевіряє кожен крок і не дає нічого зламати насправді. Але справжня перевірка — після екзамену: відкрий Git Bash у тестовій папці та повтори весь наскрізний сценарій курсу — гілка, два коміти, merge, push — руками, без підказок з екрана.

Чотири задачі — на кожну з цих навичок.

Модель — не єдине, що живе в diff. Файли звіту (PBIR) — це теж «рецепт», і їхній diff читається як «що зміниться на сторінці». Спробуй:

Процес команди і Fabric Git sync

Повний цикл релізу треба знати напамʼять — саме він повторюється в команді щодня.

Цільвідтворити повний цикл релізу по памʼяті і правильно діяти в типових ситуаціях Fabric Source control

Нагадаємо командний потік: робота йде в гілці feature/*, там накопичуються коміти; готову фічу вливаєш у main і робиш git push; далі Git sync (Update from Git) публікує зміни в робочу область Fabric. Це і є базова публікація. Deployment pipeline (Dev→Prod) — опційний рівень зверху для команд з окремими середовищами, а не обовʼязкова частина процесу.

Спершу склади цикл із кроків — без підказок, по памʼяті.

Тепер — ситуації з життя. У Fabric на вкладці Source control можуть одночасно горіти «Update required» і «Uncommitted changes»; у PBIP-проєкті треба знати, які файли комітяться, а які лишаються локальними; а іноді в прод потрібен один-єдиний фікс з гілки, де ще лежить недороблена фіча.

Памʼятай принцип. Джерело правди — Git. Ручні правки в обхід репозиторію рано чи пізно розсинхронізують робочу область з історією.

Фінальний бос: повний цикл у тренажері

Одна задача, всі команди курсу — від гілки до push, без жодної підказки.

Цільбез підказок пройти весь цикл: гілка → два коміти → merge → push
Ти дійшов до фінального боса — усі команди в ньому ти вже проходив своїми руками. Застрягнеш — тисни Підказку, вона тут така сама чесна допомога, як і раніше.

Це найбільша задача курсу. Репозиторій уже склоновано з сервера, у робочій директорії — два змінені файли: report/pages/main/visual.json і definition/model.tmdl. Твоє завдання: створи гілку feature/report-header, зроби два окремі коміти (кожен файл — своїм комітом), повернись на main, влий фічу і запуш на сервер.

Якщо ти дійшов сюди і переміг боса — вітаємо! Ти пройшов увесь шлях: від першого git init до повного циклу фічі, який команди Power BI повторюють щодня. Гілки, коміти, злиття, конфлікти, push і публікація через Git sync більше не теорія — ти вмієш це руками. Ти готовий до реальної командної роботи з Power BI у Git.

Я щиро радий, що ти пройшов цей шлях сам, команду за командою. Тепер клавіатура — твоя, і жоден git push тебе більше не злякає.
