Модуль B — команди

Перші команди

Мінімум для старту. Кожен рядок прокоментовано, кожен термін пояснено.

Ціль урокупройти цикл init → status → add → commit → log власноруч.
git init
Перетворює звичайну папку на Git-репозиторій (створює .git).
Простими словами: «увімкнути запис історії» у цій папці.
git clone
Завантажує існуючий репозиторій із сервера на твій комп'ютер разом з усією історією.
Простими словами: «забрати собі повну копію спільного проєкту».
git init                   # новий репозиторій тут
git clone <url>         # завантажити існуючий

Перегляд

git status                 # стан файлів
git log --oneline --graph # історія + дерево
git diff                   # зміни ще не в staging
git diff --staged         # що піде в коміт

Як коміти будують ланцюг

Кожен коміт зберігає посилання на батька (попередній коміт). Послідовні коміти утворюють лінію; HEAD і гілка рухаються разом на новий tip (найновіший коміт). Гортай:

Візуалізація: лінійна історія
tip (верхівка гілки)
Найновіший коміт гілки — той, на який вказує її вказівник.
Простими словами: «остання фотографія в альбомі цієї гілки».

Фіксація

git add <файл>Додати файл у staging.
git add .Усі зміни в поточній папці й нижче.
git commit -m "…"Коміт із повідомленням (-m = message).
git commit --amendЗмінити ОСТАННІЙ коміт. Переписує SHA. Докладно — урок у модулі 04.
--amend переписує історію. Якщо коміт уже на сервері й спільний — після amend знадобиться push --force-with-lease, що може зламати копії колег.
Те саме кнопками. У VS Code панель Source Control робить це за два кліки: «+» біля файлу — це git add, поле повідомлення і кнопка Commit — це git commit -m. Під капотом виконуються ті самі команди, тому спершу зрозумій їх у терміналі — далі кнопки перестануть бути магією.

Симулятор: спробуй сам

Це тренажер прямо на сторінці. Команду вводиш текстом — система перевірить її як справжній Git. Коли розбереш синтаксис, відкрий справжній Git Bash у папці свого проєкту (модуль 01) і повтори ці ж команди насправді.
Модуль B — команди

Налаштування Git: config і .gitattributes

Перше у новому середовищі — представитися Git і узгодити кінці рядків. Для крос-платформної PBIP-команди це критично.

Ціль урокуналаштувати імʼя, редактор і кінці рядків для крос-платформної команди.
git config та його три рівні
--system — для всіх користувачів ПК; --global — для поточного користувача (усі його репозиторії); --local — лише для цього репозиторію (за замовчуванням).
Простими словами: налаштування «для всіх / для мене / для цього проєкту». Вужчий рівень перекриває ширший.
git config --global user.name "Іван Коваленко"   # ім'я автора комітів
git config --global user.email "ivan@company.com" # email автора
git config --global core.editor "code --wait"    # редактор повідомлень
git config --list --show-origin                # усі налаштування + звідки взяті
user.name та user.email — обов'язкові. Без них коміти підпишуться невідомо ким, і авторство в історії буде зламане.

Кінці рядків і .gitattributes

CRLF vs LF (кінці рядків)
Windows завершує рядок двома символами (CRLF), Linux/macOS — одним (LF). Якщо не узгодити, Git бачитиме «зміни» у всьому файлі лише через кінці рядків.
Простими словами: невидима різниця в «кінці рядка» через різні ОС роздуває діфи на порожньому місці.
.gitattributes
Файл у репозиторії, що задає правила обробки файлів: нормалізацію кінців рядків, які файли бінарні тощо. Діє на всіх однаково.
Простими словами: спільні «правила гри» для файлів, незалежно від ОС розробника.
# .gitattributes — однакова поведінка на всіх ОС
* text=auto                 # Git сам нормалізує кінці рядків
*.tmdl text eol=lf          # TMDL завжди LF у репозиторії
*.json text eol=lf
*.pbix binary               # не намагатись діфити/мерджити бінарник
Для PBIP-команди. Зафіксуйте *.tmdl text eol=lf і *.json text eol=lf — це прибирає фальшиві діфи через CRLF/LF між Windows-машинами та уникає шумних конфліктів у моделі й звіті.

Гілки та злиття (branch · merge)

Гілка — окрема лінія роботи. Нову фічу роблять у власній гілці, потім зливають у основну.

Ціль урокустворювати гілки, зливати їх і читати граф історії.
merge (злиття)
Об'єднання змін однієї гілки в іншу.
Простими словами: «звести напрацювання з паралельного всесвіту назад у головний».
parent (батьківський коміт)
Коміт, на якому базується інший. Merge-коміт має двох батьків.
Простими словами: «попередній кадр», з якого виріс цей.
git branch                       # список гілок
git switch -c feature/svg-badge   # створити й перейти (-c = create)
git switch main                 # перейти на існуючу
git merge feature/svg-badge     # влити в поточну
git branch -d feature/svg-badge  # видалити після злиття

Візуалізація: створення гілки й розгалуження

Гілкування крок за кроком

Два типи злиття

Fast-forward (швидке перемотування)
Коли основна гілка не отримала нових комітів після відгалуження — Git просто пересуває її вказівник уперед. Нового коміту немає.
Простими словами: «наздогнати вперед», бо розходитись не було куди.
Fast-forward merge
Merge-коміт (три-сторонній merge)
Коли обидві гілки мали власні коміти — Git створює окремий коміт із ДВОМА батьками, фіксуючи факт об'єднання.
Простими словами: «вузол», де дві лінії історії знову сходяться в одну.
Three-way merge (merge-коміт)

Як це читати у git log --graph

Реальна історія з розгалуженням і злиттям
switch vs checkout. Стара універсальна команда — git checkout. Нові розділені: git switch (гілки), git restore (файли).
Створи в реальному репозиторії гілку для експерименту: git switch -c feature/назва-експерименту, зроби пробну правку і закомить її. Потім git log --oneline --graph покаже нову лінію поруч з main. Симулятор нижче — лише розминка, звичка формується руками.

Під капотом: гілки, HEAD і detached HEAD

Гілка в Git — це не копія файлів, а крихітний вказівник. Зрозумівши це, легко не загубити коміти.

Ціль урокурозуміти, що таке HEAD, і не губити коміти в detached-стані.
Гілка = вказівник (41 байт)
Технічно гілка — це маленький файл у .git/refs/heads/, що містить лише хеш коміту (40 символів + перенесення рядка).
Простими словами: гілка — це «закладка» на коміт, а не окрема копія проєкту. Тому створення гілки миттєве й майже безкоштовне.
Файл .git/refs/heads/main містить рядок на кшталт a1b2c3d…
HEAD
Вказівник на те, де ти зараз. Зазвичай: HEAD → гілка → коміт.
Простими словами: «ти тут». Коли комітиш, рухається та гілка, на яку дивиться HEAD.
Detached HEAD (відчеплений HEAD)
Стан, коли HEAD указує ПРЯМО на коміт, а не на гілку. Виникає після git checkout <sha>.
Простими словами: ти стоїш «на коміті без закладки». Нові коміти тут не належать жодній гілці й при перемиканні можуть загубитись.
Як виникає й вирішується detached HEAD
git checkout a1b2c3d        # перейти на коміт → detached HEAD
# … зробив коміт X (поки нічий) …
git switch -c rescue         # закріпити X гілкою rescue — у безпеці

Навіщо туди йти свідомо: подивитись стару версію звіту

Detached HEAD — не тільки пастка, а й корисний інструмент. Бізнес питає: «а як ця міра рахувалась місяць тому?» Замість шукати стару копію .pbix — подорож у часі:

git log --oneline --since="1 month ago"   # знайти коміт місячної давнини
git checkout 7f3e21a                     # стати на нього (detached HEAD — це ОК)
# відкрити .pbip у Power BI Desktop — бачиш звіт таким, яким він був тоді
git switch main                          # повернутись у сьогодні

Поки ти лише дивишся й нічого не комітиш — detached HEAD повністю безпечний. Так само можна ставати на тег релізу: git checkout v3.0.

Як розпізнати, що ти в detached HEAD

  • Git Bash пише замість назви гілки SHA у дужках: ((7f3e21a…)), а після checkout — попередження You are in 'detached HEAD' state.
  • VS Code показує SHA замість імені гілки в лівому нижньому куті статус-бара.
  • Sourcetree у графі історії не підсвічує жодну гілку як поточну.
Якщо вже перемкнувся й «втратив» коміт: знайди його SHA в git reflog і створи на ньому гілку: git branch rescue <sha>. Покроковий порятунок — в уроці про reflog у модулі 04.
Модуль B — команди

Локальний і хмарний репозиторій

Найважливіша для команди тема. Тут усе показано візуально: що на твоєму комп'ютері (локально) і що на сервері (хмара) одночасно.

Ціль урокурозрізняти локальний репозиторій і сервер; знати clone, push, pull, fetch.
Локальний vs віддалений (remote)
Локальний репозиторій — на твоєму комп'ютері. Віддалений (remote) — на сервері. Це ДВІ окремі копії, що синхронізуються командами.
Простими словами: твій робочий стіл і спільна хмарна папка. Файли там не оновлюються самі — ти явно відправляєш і забираєш зміни.
origin / main / origin/main
origin — назва сервера. main — твоя локальна основна гілка. origin/main — локальне «відображення» серверної гілки на момент останнього обміну (не сам сервер у реальному часі).
Простими словами: origin/main — це «фотографія» стану сервера, яку ти останній раз бачив. Щоб оновити фото — роби fetch.

git clone — отримати копію вперше

Завантажує весь віддалений репозиторій локально й під'єднує origin.

Візуалізація: clone (сервер → твій комп'ютер)
Звідки взяти URL. В Azure DevOps на сторінці репозиторію є кнопка Clone: вона показує адресу для git clone (скопіював — і команда готова) і одразу пропонує варіант «Clone in VS Code».

git fetch / git pull — забрати чужі зміни

fetch vs pull
fetch завантажує коміти з сервера, але НЕ чіпає твої робочі файли. pull = fetch + одразу merge у поточну гілку.
Простими словами: fetch — «подивитись, що нового на сервері». pull — «подивитись і одразу собі забрати».
Візуалізація: pull (сервер ↔ твій комп'ютер)

git push — відправити свої зміни

Візуалізація: push (твій комп'ютер → сервер)
Кнопковий еквівалент. У Sourcetree це кнопки Pull і Push на верхній панелі: цифра на Pull показує, скільки чужих комітів чекає на сервері, на Push — скільки твоїх ще не відправлено. Клік виконує той самий git pull чи git push.
Якщо репозиторій починався з git init, а не clone. Локальний репозиторій ще не знає, куди пушити — origin у ньому не налаштований. Один раз під'єднай сервер: git remote add origin <url>. Пропустиш цей крок — git push відповість fatal: No configured push destination.

Коли двоє пушать одночасно — конфлікт синхронізації

Найчастіша ситуація в команді з двох розробників: обидва зробили коміти й намагаються запушити. Хто перший — той пройде. Другому push відхилять.

Візуалізація: push відхилено → pull --rebase → push
git push
# ! [rejected] main -> main (non-fast-forward)
git pull --rebase      # забрати чужий коміт і перенести свій поверх
git push               # тепер проходить
Чому --rebase, а не звичайний pull? Звичайний pull створив би merge-коміт «Merge branch main». pull --rebase переносить твій коміт поверх чужого, лишаючи історію лінійною й чистою.
git remote -v          # показати під'єднані сервери
git push -u origin feature/x  # перший push нової гілки (-u запам'ятає зв'язок)
Маєш підключений origin? Зроби реальний git push для свого останнього коміту й онови сторінку репозиторію в браузері — побачиш цей коміт нагорі історії. Якщо сервер ще не підключений — спершу git remote add origin <url>.

Автентифікація: SSH-ключі та Personal Access Token

Сервер має знати, що push робиш саме ти. Два стандартні способи — токен для HTTPS і SSH-ключі.

Ціль урокуналаштувати доступ до сервера без введення пароля щоразу.
Personal Access Token (PAT)
Згенерований на сервері (Azure DevOps, GitHub) рядок-токен, що використовується ЗАМІСТЬ пароля при роботі через HTTPS. Має термін дії та обмежені права (scopes).
Простими словами: «тимчасовий пароль лише для Git» — вводиш один раз при першому push, далі його запам'ятовує менеджер креденшлів.
Azure DevOps: User settings → Personal access tokens → New Token → права Code (Read & write).
Credential manager (менеджер креденшлів)
Системне сховище, куди Git кладе введений токен, щоб не питати його щоразу.
Простими словами: «запам'ятовувач паролів» для Git. На Windows ставиться разом із Git (Git Credential Manager).
SSH-ключі
Пара файлів: приватний ключ (лишається тільки в тебе) і публічний (додається у профіль на сервері). Сервер звіряє пару — пароль не потрібен взагалі.
Простими словами: «замок і ключ»: публічний ключ-замок віддаєш серверу, приватним ключем відмикаєш. Приватний ключ нікому й ніколи не передається.
bash
$ ssh-keygen -t ed25519 -C "ivan@company.com" # створити пару ключів $ cat ~/.ssh/id_ed25519.pub # показати ПУБЛІЧНИЙ ключ ssh-ed25519 AAAA… ivan@company.com # цей рядок → у профіль на сервері
HTTPS + PATПростіше стартувати; токен має термін дії; зберігається credential manager-ом.
SSHРазове налаштування; без паролів назавжди; URL виду git@…
Токен = пароль. Ніколи не встав-ляй PAT у код, .tmdl, M-вирази чи README — це витік (кейс 26 у довіднику). Якщо токен потрапив у коміт — негайно відкликати і згенерувати новий.
Модуль B — команди

Корисні команди: stash, cherry-pick, tag, reflog

Чотири інструменти, що часто рятують у щоденній роботі.

Ціль урокумати під рукою status, stash, tag та інші щоденні команди.
stash (схованка)
Тимчасово прибирає незакомічені зміни «у кишеню», щоб перемкнути гілку з чистою робочою директорією.
Простими словами: «згорнути недороблене вбік, щоб не заважало», потім розгорнути назад.
git stash                    # сховати зміни
git stash pop                # повернути назад
git stash list               # список схованок
cherry-pick
Копіює ОДИН конкретний коміт з іншої гілки в поточну (як новий коміт із новим SHA), не зливаючи всю гілку.
Простими словами: «зірвати одну вишеньку» — узяти лише потрібний коміт, а не весь кошик.
git cherry-pick <sha>
Візуалізація: cherry-pick одного коміту
tag (мітка)
Нерухома позначка на коміті, зазвичай для релізної версії.
Простими словами: «закладка» на важливому коміті. На відміну від гілки, тег не рухається.
git tag v3.0                # мітка на поточний коміт
git tag -a v3.0 -m "Реліз 3.0"  # анотована мітка
reflog (журнал HEAD)
Показує всі переміщення HEAD за ~90 днів. Рятує після «втрати» комітів через reset/rebase.
Простими словами: «чорна скринька» Git — записує кожен твій крок, навіть якщо коміт уже не видно.
git reflog                   # історія руху HEAD
git reset --hard <sha-з-reflog>  # відновити стан
reflog — головне рятівне коло Git. Навіть orphaned-коміти після reset --hard зазвичай ще доступні тут. Повний сценарій порятунку крок за кроком — в окремому уроці модуля 04.
Модуль B — команди

Поглиблено: вибірковий staging, diff і log

Інструменти для чистих, атомарних комітів і точного перегляду змін.

Ціль урокузбирати атомарні коміти та точно шукати зміни в історії.
git add -p (patch mode)
Додає у staging не весь файл, а окремі БЛОКИ змін (hunks) на твій вибір.
Простими словами: якщо в одному файлі дві різні правки — можна застейджити лише одну, щоб коміт лишився логічним.
Атомарний коміт
Коміт, що містить одну логічну зміну.
Простими словами: «один коміт — одна думка». Легше читати історію, робити revert і review.
git add -p            # для кожного блоку: y (взяти), n (пропустити), s (розбити на менші)
# приклад блоку у Sales.tmdl:
# - SUM(Sales[Amount])
# + SUMX(Sales, Sales[Amount] * (1 - Sales[Discount]))

diff — точний перегляд

git diffНезакомічені зміни (working dir vs staging).
git diff --stagedЩо увійде в коміт (staging vs останній коміт).
git diff --statЛише статистика: які файли й на скільки рядків змінились.
git diff --word-diffПідсвічування змінених СЛІВ, а не рядків — зручно для DAX.
git diff main..featureРізниця між двома гілками.
Щоденний сценарій. Змінив міру в Power BI Desktop, зберіг PBIP-проєкт — і ще до коміту клікаєш по файлу в Sourcetree: він показує той самий git diff, лише з підсвіткою. Побачив зайве — прибрав; усе чисто — комітиш і пушиш.

log — пошук в історії

git log --oneline --graphКомпактна історія з деревом гілок.
git log --author="Іван"Коміти конкретного автора.
git log -S "Total Sales"Коміти, де з'явився або зник рядок із цим текстом (pickaxe).
git log --follow tables/Sales.tmdlІсторія файлу, навіть якщо його перейменовували.
git show <sha>Повні деталі та зміни конкретного коміту.
PBIP-сценарій. git log -S "'Total Sales'" швидко знаходить коміт, де міру додали чи змінили; --word-diff робить зміни в DAX-формулах наочними.

git blame: хто і коли змінив цей рядок

git blame
Для кожного рядка файлу показує SHA коміту, автора й дату останньої зміни цього рядка.
Простими словами: підписи на полях зошита — біля кожного рядка видно, хто його вписав.
Бачиш у Sales.tmdl дивний фільтр у мірі — git blame одразу каже, чий це коміт і коли він зʼявився.

Типове розслідування аналітика: «минулого тижня міра рахувала інакше — хто чіпав?»

git blame tables/Sales.tmdl     # біля кожного рядка: SHA, автор, дата зміни
7f3e21a (Олена 2026-06-25) SUMX(Sales, Sales[Amount] * (1 - Sales[Discount]))
git log -p -- tables/Sales.tmdl   # уся історія змін файлу з діфами
Без термінала — те саме в браузері. У веб-інтерфейсі Azure DevOps відкрий файл → History або Annotate (у GitHub — Blame): автора кожного рядка видно одразу в браузері.
