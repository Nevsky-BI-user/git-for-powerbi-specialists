Перший репозиторій для PBIP-проєкту

Три команди — і твій звіт уже під захистом історії.

Цільсамостійно, без підказок у коді, пройти шлях git init → git add → git commit на структурі PBIP-проєкту

Коротке нагадування: PBIP-проєкт — це не «чорна скринька», як .pbix, а звичайні текстові файли. report.pbip — маленький файл-вказівник на частини проєкту, а definition/model.tmdl — текстовий опис моделі: таблиці, міри, звʼязки. Саме тому Git «розуміє» PBIP: він порівнює текст рядок за рядком, як редактор порівнює дві версії документа. Теорію цих кроків ми детально розбирали в модулі 02 — тут лише практика.

Ланцюжок перших дій завжди той самий: git init створює порожній репозиторій (невидиму теку .git, де житиме вся історія), git add кладе файли у staging — «кошик» майбутнього коміту, а git commit -m "..." фіксує знімок в історії.

Як працює тренажер нижче: зліва — термінал, у який ти вводиш справжні команди Git; справа — жива схема стану репозиторію і чипи файлів, які змінюються залежно від того, де зараз файл — у робочій теці, у staging чи вже в коміті. Це навчальна симуляція прямо на сторінці: вона не чіпає файли на твоєму компʼютері і не потребує встановленого Git (віконце лише стилізоване під Git Bash). Ті самі команди потім вводитимеш у справжньому Git Bash на своєму проєкті — тренажер існує, щоб напрацювати руку безпечно. Якщо застряг, кнопка Підказка підкаже наступний крок, Показати розвʼязок розкриє всі команди, а Скинути поверне задачу на початок.

Тренажер нижче — безпечна пісочниця: кожна команда перевіряється, і зламати тут нічого не можна. Розвʼяжеш обидві задачі цієї сторінки — відкрий справжній Git Bash у тестовій папці (не в робочому проєкті!) і повтори ті самі три команди руками: звичка формується на справжній клавіатурі, а не на екрані.

Ситуація: ти щойно зберіг звіт у форматі PBIP і хочеш поставити його під контроль версій. Створи репозиторій і зроби перший коміт, у який увійдуть обидва файли проєкту.

Пиши повідомлення коміту так, щоб через місяць ти сам зрозумів, що це було: «первинна версія звіту продажів» набагато корисніше, ніж «зміни».

Staging під контролем: осмислені коміти

Staging — це фільтр між «що я наробив» і «що піде в історію».

Цількерувати staging свідомо: окремі коміти для окремих змін і коміт лише частини правок

За день роботи над звітом у робочій теці накопичується все підряд: виправлена міра, нова таблиця, випадково зачеплений службовий файл. Staging дозволяє не звалювати це в один коміт «усе разом», а розкласти по поличках: git add конкретного файлу кладе в кошик лише його, а git status щоразу чесно показує, що вже в кошику, а що ще ні. Самі команди — у модулі 03.

Ситуація для першого тренажера: ти змінив міру в definition/model.tmdl і додав нову таблицю Sales. Це дві різні зміни, і в історії вони мають жити окремо — зроби два осмислені коміти.

Чому це важливо? Дрібні осмислені коміти — це історія, яку можна читати: у git log видно «виправив міру маржі», «додав таблицю Sales», і кожен крок можна окремо переглянути чи відкотити. Один великий коміт «зміни за вівторок» — це валіза без ярликів: коли зламається міра, доведеться перетрушувати все разом.

Другий тренажер — про вибірковість: закомітити треба лише зміни моделі у model.tmdl, а от diagramLayout.json — службовий файл розкладки діаграми — має лишитися незакоміченим.

Звичка писати git add . «на автоматі» — головний ворог чистої історії: у коміт залітає все підряд, включно з файлами, які ти туди не збирався класти. Спершу git status, потім — адресний git add.

Схема: чому історія — це ланцюг

Кожен коміт тримає за руку попередній — тому історію не можна непомітно переписати.

Цільчитати схему комітів: хто чий батько і чому історію не можна «редагувати в середині»

Два терміни простими словами. Батько коміту — це коміт, з якого виріс поточний: кожен новий знімок запамʼятовує, після кого він зʼявився, як сторінка щоденника «знає» попередню сторінку. SHA — унікальний підпис коміту (оті a1b2c3d у git log): він обчислюється і з вмісту коміту, і з підпису його батька. Тому змінити щось «у середині» непомітно неможливо: старий коміт отримав би інший підпис, а отже «зламалися» б підписи всіх його нащадків.

Пройди схему нижче по кроках: спочатку — як коміти шикуються в ланцюг батьків, потім — чому вставити коміт у середину означає перебудувати весь хвіст історії після нього.

Ланцюг комітів

Для Power BI це означає просте правило: історія моделі — надійний журнал. Якщо тиждень тому в коміт потрапила міра з помилкою, цей факт не «підправиш заднім числом» — можна лише додати зверху новий коміт із виправленням.

Екзамен-розминка: читаємо вивід Git

Git завжди каже, що станеться далі — треба лише вміти його читати.

Цільпо чистому виводу git status і git log розуміти, що станеться далі

Екзаменаційні питання складніші за міні-перевірки в модулях: тут немає переказу теорії — є справжній вивід терміналу, який треба прочитати так, як ти робитимеш це в реальній роботі. Дивись на git status і git log --oneline уважно: кожен рядок — це підказка, що саме Git зробить наступною командою. Не поспішай з відповіддю, поки не зрозумів кожен рядок виводу.
