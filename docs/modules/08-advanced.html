Модуль G — просунуті техніки

git bisect: знайти коміт, що вніс баг

Бінарний пошук по історії: замість перевіряти десятки комітів — лише кілька кроків.

Ціль урокузнаходити коміт-винуватець бінарним пошуком.
git bisect
Ділить діапазон комітів навпіл: ти позначаєш поточний як good чи bad, Git звужує діапазон, доки не знайде перший «поганий» коміт.
Простими словами: гра «вгадай число»: Git щоразу ділить навпіл і питає «тут уже зламано?», швидко звужуючи до винного коміту.
Бінарний пошук винного коміту
good (баг відсутній)bad (баг є)невідомо / тест
git bisect start
git bisect bad            # поточний коміт зламаний
git bisect good a1b2c3d   # старий коміт ще робочий
# Git робить checkout середнього коміту — тестуєш — і кажеш:
git bisect good   # або   git bisect bad
# … повторюєш, доки: "<sha> is the first bad commit"
git bisect reset          # повернутися на свою гілку
PBIP-сценарій. Звіт «раптом» порахувався інакше — bisect швидко вкаже коміт, де змінилась міра чи зв'язок, навіть серед сотень комітів.
Як це виглядає на практиці. bisect — територія термінала Git Bash: у Sourcetree такої команди немає. Зате перевірка кожного кроку — звична: Git уже підмінив файли на диску, тож відкриваєш PBIP у Power BI Desktop, дивишся, чи є баг, закриваєш файл і відповідаєш good чи bad.

Git hooks: автоматичні перевірки

Hook — локальний скрипт, що сам спрацьовує на певну подію Git.

Ціль урокуавтоматизувати перевірки перед комітом і пушем.
Git hook
Скрипт у .git/hooks/, який Git запускає автоматично перед або після події (коміт, push).
Простими словами: «правило-автомат»: напр. не дати закомітити зламаний TMDL.
pre-commitПеред комітом. Для PBIP: перевірити синтаксис TMDL/JSON, заблокувати секрети.
commit-msgПеревірити формат повідомлення (Conventional Commits).
pre-pushПеред push. Напр. запустити Best Practice Analyzer.
# .git/hooks/pre-commit (спрощено)
# 1) перевірити, що змінені .json валідні; інакше — стоп-коміт
# 2) перевірити відсутність шаблонів секретів
Де це писати. Hook — звичайний скрипт: зручно писати й редагувати його у VS Code (з підсвіткою синтаксису), а перш ніж класти в .git/hooks/ — прогнати вручну в терміналі Git Bash і переконатися, що він працює.
Команді. Локальні hooks не потрапляють у .git репозиторію; щоб поширити їх, тримають скрипти в самому репозиторії (напр. папка hooks/) і підключають, або переносять перевірки в CI (Azure Pipelines) на сервері.
Модуль G — просунуті техніки

Worktrees: кілька гілок одночасно на диску

Терміновий hotfix без stash і без переривання поточної роботи.

Ціль урокуробити терміновий hotfix, не чіпаючи поточну роботу.
git worktree
Дозволяє мати кілька робочих директорій з одного репозиторію — кожна на своїй гілці одночасно.
Простими словами: «друге вікно» того самого проєкту на іншій гілці: правиш hotfix у сусідній папці, не чіпаючи незакомічені зміни в основній.
git worktree add ../project-hotfix hotfix/refresh  # нова папка на гілці hotfix
# … правиш і комітиш hotfix у ../project-hotfix …
git worktree list                                # перелік робочих директорій
git worktree remove ../project-hotfix             # прибрати, коли готово
Альтернатива stash. Замість «згорнути все вбік» — відкрити hotfix в окремій папці; основна директорія з недоробленою фічею лишається недоторканою. Команди worktree теж живуть у терміналі Git Bash, а далі все як завжди: PBIP із сусідньої папки відкриваєш у Power BI Desktop, файли правиш у VS Code.
