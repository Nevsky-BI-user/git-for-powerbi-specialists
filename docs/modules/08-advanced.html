Інструменти на виріст

Три інструменти на випадок «колись знадобиться» — не щоденна робота, а розширення набору на крайній випадок.

Ціль урокузнати, що ці інструменти існують, і впізнати момент, коли варто по них повернутися.

Чесно: інструментам нижче більшість PBI-розробників не знайде застосування роками. Мета цього розділу — не вивчити їх напам'ять, а знати, що вони існують, і впізнати ситуацію, коли варто повернутися сюди за рецептом. І одразу підказка на найчастіший випадок: якщо питання просте — «коли саме зламалась ця міра?» — зазвичай досить одного рядка git log -S "назва міри" з модуля 03 (готовий рецепт є і в шпаргалці, розділ «Типові сценарії»).

Якщо ти дійшов до цього модуля — базу вже маєш: коміти, гілки, rebase і PBIP у Git для тебе вже рутина. Три інструменти нижче — не обов'язкова програма, а «на всякий випадок»: досить впізнати ситуацію, коли вона трапиться.

git bisect — двійковий пошук зламаного коміту

git bisect ділить діапазон комітів навпіл: ти щоразу кажеш Git, чи в середньому коміті баг уже є (bad) чи ще нема (good), і за кілька кроків замість перевірки десятків комітів підряд знаходиш винуватця. Реально береш його в руки рідко — і головно тоді, коли git log -S "назва міри" НЕ допоміг: сама назва чи текст формули в комітах не змінювались, а результат все одно зʼїхав (річ у звʼязку, фільтрі чи іншій мірі, від якої ця залежить). Git сам перемикає робочу папку на черговий середній коміт — тобі лишається відкрити .pbip і подивитись, чи баг там є.

git bisect start
git bisect bad
git bisect good a1b2c3d
git bisect reset

Git hooks — автоматичні перевірки перед комітом

Git hook — скрипт у .git/hooks/, який Git запускає сам на певну подію: перед комітом (pre-commit), перед push (pre-push) і так далі. Чесно: це вже територія shell-скриптів, а не звичних Git-команд, і в щоденній роботі PBI-розробника вона майже не зʼявляється — таке налаштовують один раз, зазвичай хтось досвідченіший у команді, а решта просто користується готовим захистом. Типове застосування — не пустити в коміт закомічений cache.abf чи localSettings.json.

#!/bin/sh
# стоп-коміт, якщо у staged потрапив кеш або локальні налаштування
if git diff --cached --name-only | grep -qE "cache\.abf|localSettings\.json"; then
  echo "У коміті службові файли — прибери їх"
  exit 1
fi

git worktree — друга робоча папка

git worktree дає другу робочу папку того самого репозиторію — кожна на своїй гілці одночасно: правиш терміновий hotfix у сусідній директорії, не чіпаючи незакомічену роботу в основній. Чесно: у 90% випадків для цього вистачає git stash — сховав зміни, перемкнув гілку, зробив хотфікс, повернув назад. worktree виправдовує себе рідше: коли stash незручний (посеред багатоденного рефакторингу з купою недооформлених правок) або коли треба звірити дві гілки одночасно, відкривши обидві папки в Power BI Desktop поруч.

git worktree add ../project-hotfix hotfix/refresh
