Модуль C — виправлення

Скасування змін

Три інструменти для трьох ситуацій. Плутанина між ними — найчастіша причина втрати роботи. Усі три доступні і з Git Bash, і кнопками у Sourcetree чи VS Code — але спершу розберися, що саме робить кожен.

Ціль урокуобирати правильний інструмент скасування без втрати роботи.
orphaned commit (осиротілий коміт)
Коміт, на який більше не вказує жодна гілка чи тег. Видимо «зник», але фізично ще лежить у .git деякий час.
Простими словами: фотографія, вирвана з альбому — її ще можна знайти в смітнику (reflog), поки не прибрали.

Як reset рухає вказівник (граф)

Візуалізація: reset пересуває гілку назад

Три режими reset (що зі зонами)

Граф історії при reset завжди однаковий. Різниця — що відбувається зі staging і робочою директорією. Обери режим:

HEAD / гілка
—
Staging
—
Working Dir
—
Обери режим, щоб побачити які зони він зачіпає.
Де це у Sourcetree: правий клік по коміту в графі → «Reset current branch to this commit», у діалозі обираєш режим Soft / Mixed / Hard. Це та сама команда, що й у Git Bash: кнопка не робить її безпечнішою, тож перед Hard так само переконайся, що незакомічені зміни тобі більше не потрібні.
git restore <файл>Відкинути зміни файлу у working dir. Історію не чіпає.
git restore --staged <файл>Прибрати файл зі staging (зміни лишаються).
git reset --soft <commit>Гілка назад; зміни лишаються у staging.
git reset --mixed <commit>За замовч. Гілка назад + очистка staging; зміни у working dir.
git reset --hard <commit>Небезпечно. Усі три зони назад. Незакомічене втрачається.
git revert <commit>НОВИЙ коміт, що скасовує зміни. Безпечно для спільних гілок.

Спробуй сам: коміт зроблено зарано — відкоти його, але всі зміни збережи.

Де це у VS Code: «Discard Changes» (іконка ↶ біля файлу в панелі Source Control) — це і є git restore. Типовий сценарій: зіпсував міру у звіті в Power BI Desktop, зберіг PBIP, глянув diff — і одним кліком повернув файл до стану останнього коміту. Огляд самих клієнтів — в уроці «Sourcetree та інші графічні клієнти».

А тепер найчастіша щоденна ситуація: файл зіпсовано ще до коміту.

reset переписує історію, revert — ні. На спільних гілках (main) — лише revert. reset --hard — локально й усвідомлено. Три наступні уроки розбирають revert, reflog і amend докладно.
Модуль C — виправлення

git revert докладно: скасування без переписування

Єдиний інструмент скасування, який безпечний для гілки, що вже на сервері. Розберемо, як він працює й де його підводні камені.

Ціль урокускасовувати запушені коміти через revert і розуміти, чому це не reset.
git revert
Створює НОВИЙ коміт, зміни якого дзеркально протилежні вказаному коміту. Історія не переписується — вона росте вперед.
Аналогія: бухгалтерія. Помилкову проводку не виривають із журналу — додають сторнувальний запис. Обидва рядки видно, баланс виправлено.

Типовий сценарій у Power BI

Ти виправив міру, закомітив, запушив — а наступного ранку зʼясувалось, що новий фільтр ламає квартальний підсумок. Коміт уже в main, робоча область уже синхронізувалась. Переписувати історію пізно — скасовуємо коміт новим комітом:

git log --oneline -3          # знайти SHA проблемного коміту
e4f5a6b fix: correct YTD filter   ← оцей ламає підсумок
git revert e4f5a6b            # Git створює коміт "Revert …" з протилежними змінами
git push                       # колеги отримають скасування звичайним pull

Після push робоча область підтягне скасування через Git sync — звіт повернеться до робочої версії, а в історії буде видно і помилку, і виправлення.

Що варто знати наперед

  • Revert може конфліктувати. Якщо після проблемного коміту той самий рядок TMDL змінювали ще раз, Git попросить вирішити конфлікт — так само, як при merge (модуль 05): виправив у VS Code → git add → git revert --continue.
  • Revert merge-коміту — окремий випадок: потрібен ключ -m 1, щоб указати, «до якого з батьків» повертатись. Новачку простіше скасувати окремі коміти або покликати досвідченого колегу.
  • Кілька комітів скасовуються по одному (git revert <старий> <новіший>) — кожен отримає свій коміт-скасування.
Де це у GUI: у Sourcetree — правий клік по коміту → Reverse commit; у VS Code з розширенням GitLens — Revert Commit у контекстному меню історії. Під капотом — той самий git revert.
Модуль C — виправлення

git reflog: рятівне коло, коли «все зникло»

Майже нічого в Git не зникає безслідно. Навіть після reset --hard коміт можна дістати — якщо знати, де шукати.

Ціль урокувідновлювати «втрачені» коміти через reflog після reset, rebase чи detached HEAD.
reflog (reference log)
Локальний журнал усіх переміщень HEAD: кожен commit, switch, reset, rebase, merge лишає рядок із SHA. Зберігається близько 90 днів.
Аналогія: чорна скринька літака. Що б не сталося з історією — тут записано, де ти був на кожному кроці, і туди можна повернутись.

Сценарій порятунку крок за кроком

Зробив git reset --hard HEAD~2, щоб «прибрати сміття», — і разом зі сміттям зніс коміт із готовою сторінкою звіту. У git log його більше немає. Рятуємо:

git reflog
3aa81f2 HEAD@{0}: reset: moving to HEAD~2     ← сам reset
e4f5a6b HEAD@{1}: commit: feat: OKR page      ← ось зниклий коміт!
git branch rescue e4f5a6b   # закладка на «зниклий» коміт — тепер він у безпеці
git switch rescue            # перевір вміст: сторінка на місці

Далі — як зі звичайною гілкою: злити rescue у main або забрати з неї окремі коміти через cherry-pick.

Коли reflog рятує

  • Після reset --hard не туди — найчастіший випадок.
  • Коміт зроблено в detached HEAD і «загублено» перемиканням гілки.
  • Rebase пішов не так — git reflog покаже стан до нього (модуль 05).
  • Випадково видалив гілку — SHA її верхівки досі в reflog.
Три обмеження. Reflog суто локальний — журнал колеги з його компʼютера не подивишся. Він не вічний: стандартно записи живуть ~90 днів. І незакомічені зміни (які не встигли стати комітом) reflog не поверне — тому вигідно комітити часто.
Модуль C — виправлення

git commit --amend: відполірувати останній коміт

Одруківка в повідомленні або забутий файл — не привід для другого коміту «fix typo». Останній коміт можна переробити.

Ціль урокувиправляти останній коміт (повідомлення чи вміст) і знати, коли amend уже заборонений.
git commit --amend
Замінює останній коміт новим: те саме місце в історії, але оновлені вміст і/або повідомлення. SHA коміту змінюється.
Аналогія: переклеїти останню сторінку в альбомі, поки нікому його не показував. Сторінка нова — але для всіх, хто альбом уже бачив, він тепер «не той».

Два щоденні випадки

# 1) одруківка в повідомленні
git commit --amend -m "fix: correct YTD filter in Sales"

# 2) забув файл: зберіг у Power BI Desktop лише модель, а зміну сторінки — ні
git add report/definition/pages/okr.json
git commit --amend --no-edit   # файл доїхав тим самим комітом, повідомлення без змін
Головне правило: amend — лише до push. Amend переписує SHA, тобто історію. Якщо коміт уже на сервері, після amend push вимагатиме --force, а це ламає копії колег — та сама логіка, що й у золотому правилі rebase (модуль 05). Коміт уже запушено? Виправляй наступним комітом або revert-ом.
Де це у GUI: у Sourcetree під полем повідомлення є прапорець Amend latest commit; у VS Code — команда «Commit (Amend)». Обидва роблять той самий git commit --amend.
