Модуль C — виправлення

Скасування змін

Три інструменти для трьох ситуацій. Плутанина між ними — найчастіша причина втрати роботи.

Ціль урокуобирати правильний інструмент скасування без втрати роботи.
orphaned commit (осиротілий коміт)
Коміт, на який більше не вказує жодна гілка чи тег. Видимо «зник», але фізично ще лежить у .git деякий час.
Простими словами: фотографія, вирвана з альбому — її ще можна знайти в смітнику (reflog), поки не прибрали.

Як reset рухає вказівник (граф)

Візуалізація: reset пересуває гілку назад

Три режими reset (що зі зонами)

Граф історії при reset завжди однаковий. Різниця — що відбувається зі staging і робочою директорією. Обери режим:

HEAD / гілка
—
Staging
—
Working Dir
—
Обери режим, щоб побачити які зони він зачіпає.
git restore <файл>Відкинути зміни файлу у working dir. Історію не чіпає.
git restore --staged <файл>Прибрати файл зі staging (зміни лишаються).
git reset --soft <commit>Гілка назад; зміни лишаються у staging.
git reset --mixed <commit>За замовч. Гілка назад + очистка staging; зміни у working dir.
git reset --hard <commit>Небезпечно. Усі три зони назад. Незакомічене втрачається.
git revert <commit>НОВИЙ коміт, що скасовує зміни. Безпечно для спільних гілок.
reset переписує історію, revert — ні. На спільних гілках (main) — лише revert. reset --hard — локально й усвідомлено.
