Модуль F — командна робота

.gitignore для PBIP

.gitignore — список шляхів, які Git навмисно ігнорує. Для PBIP критично: деякі файли локальні або важкі.

Ціль урокутримати репозиторій чистим: що ігнорувати і чому.
.gitignore
Текстовий файл у репозиторії зі списком шаблонів шляхів, які Git не відстежує й не комітить.
Простими словами: «список того, що не зберігати в історії» — тимчасове, важке чи особисте.

Що Power BI Desktop створює автоматично

# Дефолтний .gitignore від Power BI Desktop:
**/.pbi/localSettings.json
**/.pbi/cache.abf

**/ = «у будь-якій підпапці».

Розширений варіант для команди

# Локальний кеш і налаштування Power BI
**/.pbi/
*.pbix
*.pbit
# Сміття ОС та редакторів
.DS_Store
Thumbs.db
.vscode/
cache.abfЛокальна копія моделі з даними (ABF = Analysis Services Backup). Важка, машинозалежна; спричиняє «type conflict» між ПК.
localSettings.jsonНалаштування лише для поточного користувача й комп'ютера.
*.pbix / *.pbitБінарники. У PBIP-проєкті не потрібні в репозиторії.
ОБОВ'ЯЗКОВО комітити: .tmdl, вміст definition/, сам .pbip і .gitignore.
Модуль F — командна робота

Стратегія гілок для PBIP-команди

Робоча модель: одна інтеграційна гілка main, feature-гілки, а публікація йде не через гілку, а через робочу область (Git sync); окремий Prod, якщо він є, досягається через deployment pipeline.

Ціль урокупрацювати короткими feature-гілками без хаосу.

main

Інтеграційна гілка. Сюди зливають feature-гілки. Під'єднана до Dev робочої області через Git integration.

feature/*

Гілка під кожну зміну. Від main → merge назад у main після перевірки проєкту в Power BI Desktop.

Dev / Prod

Це робочі області (workspaces), НЕ гілки Git. main синхронізує робочу область; якщо середовищ два, deployment pipeline промоутить Dev → Prod (опційний рівень).

hotfix/*

Термінове виправлення; так само зливається в main і публікується через Git sync (у схемі з Dev/Prod — далі pipeline).

Pull Request (PR) / Merge Request
Запит на злиття гілки в основну з переглядом колегою перед прийняттям.
Простими словами: «прошу влити мою гілку — гляньте перед цим».
У цій PBIP-команді детальний review діфів у PR непрактичний (TMDL/JSON-шум, без прев'ю звіту), тож зміни зливають у main напряму; роль бар'єру виконують валідація в Desktop, CI та (за наявності Dev/Prod) deployment pipeline. Детальніше — урок «Робочий процес: merge → деплой».
CI (Continuous Integration)
Автоматичні перевірки, що запускаються на сервері при push (напр. валідація моделі, Best Practice Analyzer).
Простими словами: «робот-контролер», що перевіряє зміни без участі людини.

Потік фічі у графі

feature → merge у main
Fabric / Power BI Git integration. Гілку main прив'язують до робочої області. Після push у main роблять Update from Git — звіт опубліковано; для звичайної публікації цього достатньо. Команди з окремими Dev і Prod додають другий етап: deployment pipeline промоутить Dev → Prod.
Модуль F — командна робота

Конфлікти у PBIP-файлах

PBIP діфиться як текст, але має специфіку. Де найчастіше виникають конфлікти й як читати.

Ціль урокувирішувати конфлікти TMDL/JSON і валідувати результат.
tables/*.tmdlДві гілки змінили міри в одній таблиці. Конфлікт у блоці measure чи порядку об'єктів.
lineageTagРегенеровані GUID розійшлися. Лишити будь-який валідний унікальний tag.
relationships.tmdlПаралельно додані зв'язки. Об'єднати обидва блоки relationship.
PBIR (report JSON)Зміни позицій/властивостей візуалів. Гранулярність PBIR зменшує конфлікти.
Після вирішення TMDL-конфлікту відкрий проєкт у Power BI Desktop або провалідуй у Tabular Editor. Git перевіряє лише текст, не валідність моделі.
Інструмент для вирішення. Конфліктний TMDL/JSON відкривай у VS Code: вбудований merge-редактор показує «свою» та «чужу» версії поруч і дає кнопки Accept Current / Accept Incoming / Accept Both — не треба вручну вишукувати маркери <<<<<<<.

Склади повний ланцюжок: від CONFLICT до валідної моделі.

Профілактика

  • Короткоживучі feature-гілки.
  • Частий rebase на свіжий main, поки гілка маленька.
  • Перевірка моделі в Power BI Desktop перед merge у main (бо review діфів у PR непрактичний).
  • CI: автозапуск Best Practice Analyzer (Tabular Editor) на .SemanticModel при push.
Модуль F — практика

Базові сценарії: двоє розробників

Кожен крок супроводжується git-командою та схемою трьох репозиторіїв: ліворуч локальний у Розробника A, по центру — сервер (хмара), праворуч — локальний у Розробника B. Найновіший коміт — угорі. Гортай кроки.

Ціль урокувідпрацювати чотири типові командні ситуації покроково.
спільний / початковий коміткоміт Розробника Aкоміт Розробника B

Сценарій 1 — базовий цикл

Найфундаментальніше: як один коміт мандрує від A через сервер до B. Це основа, на якій тримаються всі інші сценарії.

commit → push → pull
Ті самі кроки в інструментах. Розробник A вносить зміну і зберігає проєкт у Power BI Desktop, потім у Sourcetree переглядає diff змінених TMDL/JSON-файлів, комітить і натискає Push. Розробник B натискає Pull у Sourcetree — і бачить коміт A в історії та оновлені файли у своєму проєкті.

Спробуй сам: ти — Розробник B, колега щойно запушив коміт на сервер.

Сценарій 2 — паралельна робота, push відхилено

Двоє комітять одночасно різні файли. Хто пушить другим — отримує відмову non-fast-forward і робить pull --rebase. Конфлікту немає, бо файли різні.

паралельні коміти + pull --rebase

Сценарій 3 — конфлікт того самого рядка

Найчастіший справжній конфлікт: обидва змінили ту саму міру. Покроково: від відмови до вирішення конфлікту й push. На практиці конфліктний model.tmdl відкривають у merge-редакторі VS Code й обирають або поєднують версії кнопками (як саме — урок «Конфлікти у PBIP-файлах»).

конфлікт у model.tmdl та його вирішення

Сценарій 4 — гілка для фічі

A створює feature-гілку, пушить її на сервер, B її забирає через fetch і приєднується до роботи в одній гілці.

switch -c → push -u → fetch → switch
Модуль F — практика

Сценарії середнього рівня

Складніші ситуації — теж покроково, з командами та тією ж схемою A / сервер / B.

Ціль урокувідпрацювати складніші командні ситуації.
спільний коміткоміт Aкоміт B

Сценарій 5 — злиття фічі в main

Готову гілку feature зливають у main (локально git merge або кнопкою merge у веб) і пушать; обидва розробники оновлюють свій локальний main. Це стан перед деплоєм у Dev/Prod.

PR merge → pull в обох

Сценарій 6 — випадково закомічений cache.abf

A зробив git add . і затягнув важкий cache.abf → B отримує type-conflict при відкритті. Прибираємо через git rm --cached + .gitignore.

type-conflict через кеш та його усунення

Сценарій 7 — відновлення після reset --hard

A помилково скинув два коміти командою reset --hard. Здається, що робота втрачена — але reflog усе повертає.

втрата комітів та відновлення через reflog

Сценарій 8 — терміновий hotfix через cherry-pick

Треба внести лише один коміт-фікс у prod, не зливаючи всю недороблену гілку. cherry-pick бере рівно один коміт.

cherry-pick одного коміту в prod
Модуль F — довідник

Довідник проблем PBIP-команди (1–30)

Ситуативні кейси з командним розв'язанням крок за кроком. Доповнюють базові сценарії вище. A і B — два розробники.

Ціль урокушвидко знаходити готове рішення типової проблеми.
Конфлікт моделі (TMDL)

1Дубль lineageTag — модель не відкривається

Проблема: після merge Power BI Desktop не відкриває проєкт: duplicate lineageTag.

Причина: обидві гілки додали об'єкти; при злитті два отримали однаковий GUID.

git status                      # бачимо конфлікт у tables/Sales.tmdl
# відкрити файл, замінити ОДИН lineageTag на новий унікальний GUID
git add tables/Sales.tmdl
git rebase --continue           # або git commit (для merge)

Підсумок: lineageTag має бути унікальним у межах моделі; провалідувати в Tabular Editor.

Конфлікт моделі

2Потрібні обидві логіки міри

Проблема: A зробив суму, B — зважений розрахунок однієї міри; merge дає конфлікт, але обидві версії цінні.

Причина: дві ідеї втиснуто в один об'єкт.

# у конфліктному блоці лишити ОБИДВІ як дві окремі міри:
#   measure 'Total Sales'           = SUM(...)
#   measure 'Total Sales (weighted)' = SUMX(...)
git add tables/Sales.tmdl && git rebase --continue

Підсумок: дві окремі міри замість однієї — конфлікт зникає, обидві логіки збережено.

Конфлікт моделі

3Однакова назва нової колонки

Проблема: A і B додали колонку з однаковою назвою в одну таблицю → конфлікт + дубль.

Причина: неузгоджене іменування.

# лишити один варіант; якщо потрібні обидві — перейменувати одну
git add tables/Employee.tmdl && git commit

Підсумок: узгодити іменування наперед, щоб уникнути повтору.

Конфлікт моделі

4Неоднозначні зв'язки після merge

Проблема: модель попереджає про ambiguous relationships.

Причина: обидва додали зв'язок між тими ж таблицями.

# у relationships лишити один активний зв'язок, зайвий:
#   isActive: false   (або видалити блок relationship)
git add definition/relationships.tmdl && git commit

Підсумок: між двома таблицями — лише один активний шлях.

Конфлікт моделі

5Видалена-vs-змінена таблиця

Проблема: merge показує «deleted by us / modified by them».

Причина: A видалив таблицю, B у ній щось змінив.

git status                # deleted by us: tables/Old.tmdl
# А: таблиця не потрібна — перенести зміни в актуальну, прийняти видалення:
git rm tables/Old.tmdl
# Б: потрібна — відновити:
git checkout --theirs tables/Old.tmdl && git add tables/Old.tmdl
git commit

Підсумок: свідомо вибрати — видалити чи відновити.

Конфлікт моделі

6Конфлікт у model.tmdl (культура / анотації)

Проблема: конфлікт у службових рядках culture / annotation / списку таблиць.

Причина: обидва пересохранили модель, службові рядки розійшлись.

# лишити один службовий блок; якщо мінялись таблиці — обєднати посилання
git add definition/model.tmdl && git commit

Підсумок: провалідувати модель після злиття.

Конфлікт моделі

7Залежна зміна → битий референс

Проблема: після часткового merge міра B посилається на колонку A, якої ще немає.

Причина: залежні зміни злиті не разом.

git switch main && git pull          # підтягнути колонку (A)
git switch feature && git rebase main  # тепер міра бачить колонку

Підсумок: зливати залежні зміни одним PR або в правильному порядку.

Кеш / службові файли

8localSettings.json створює шум

Проблема: у кожному PR з'являються зміни localSettings.json від іншого розробника.

Причина: персональні налаштування під контролем версій.

git rm --cached "**/.pbi/localSettings.json"
# переконатись, що в .gitignore є **/.pbi/
git commit -m "stop tracking localSettings"

Підсумок: персональні налаштування не комітяться.

Кеш / службові файли

9*.pbix роздуває репозиторій

Проблема: хтось закомітив бінарний .pbix; репо важчає з кожним комітом.

Причина: бінарник зберігається повністю при кожній зміні.

git rm --cached *.pbix
# echo "*.pbix" у .gitignore
git commit -m "ignore pbix"
# очистити історію (за потреби): git filter-repo --path-glob '*.pbix' --invert-paths

Підсумок: бінарники не тримати в репозиторії.

Кеш / службові файли

10.gitignore не діє на вже доданий файл

Проблема: файл у .gitignore, але Git далі показує його зміни.

Причина: .gitignore діє лише на ще НЕ відстежувані файли.

git rm --cached path/to/file
git commit -m "apply gitignore"

Підсумок: після rm --cached ignore починає діяти.

Форматування / інструменти

11Whitespace-churn через різні версії Tabular Editor

Проблема: у diff сотні «змін» — лише відступи й порядок рядків, логіка та сама.

Причина: A і B серіалізують TMDL різними версіями/налаштуваннями.

# узгодити версію Tabular Editor і serialization settings у команді
# один нормалізуючий коміт спільним інструментом:
git add . && git commit -m "normalize TMDL serialization"

Підсумок: однакові налаштування серіалізації прибирають шумні діфи.

Форматування / інструменти

12Битий format string через локаль

Проблема: після merge числові/SVG-міри показують кому замість крапки або некоректний формат.

Причина: у FORMAT() не задано локаль.

# задати локаль явно:
#   = FORMAT([Value], "0.00", "en-US")
git add tables/Sales.tmdl && git commit -m "fix FORMAT locale"

Підсумок: для SVG-координат і чисел завжди вказувати "en-US".

Git-механіка

13Випадковий detached HEAD

Проблема: після checkout <sha> зроблено коміти, потім перемкнулись на гілку — коміти «зникли».

Причина: робота велась без гілки (detached HEAD).

git switch -c feature/save      # закріпити коміти гілкою ДО перемикання
# якщо вже пізно:
git reflog                       # знайти SHA
git branch rescue <sha>

Підсумок: робота без гілки = ризик загубити коміти.

Git-механіка

14Конфлікт-шторм у довгій гілці

Проблема: гілка жила два тижні; при merge — десятки конфліктів.

Причина: гілка сильно відійшла від main.

git switch feature
git rebase main                 # робити ЩОДНЯ, поки гілка маленька
# велику фічу розбивати на кілька дрібних PR

Підсумок: частий rebase не дає історіям сильно розійтись.

Git-механіка

15Безлад у комітах перед PR

Проблема: у гілці 12 хаотичних комітів; review важко читати.

Причина: коміти не причесані перед PR.

git rebase -i main              # squash/fixup дрібних, reword повідомлень
git push --force-with-lease     # гілка особиста — безпечно

Підсумок: чистий список комітів спрощує review.

Git-механіка

16Конфлікт у файлі-вказівнику .pbip

Проблема: конфлікт у PDP.pbip після того, як обидва пересохранили проєкт.

Причина: службовий JSON-вказівник розійшовся.

git status                # конфлікт у PDP.pbip
# лишити одну валідну версію; перевірити перелік артефактів (Report + SemanticModel)
git add PDP.pbip && git commit

Підсумок: відкрити в Desktop для перевірки після злиття.

Конфлікт звіту (PBIR)

17Обидва пересунули той самий візуал

Проблема: конфлікт у JSON одного візуала (координати/розмір).

Причина: A і B рухали один візуал.

git checkout --ours <visual.json>   # або виставити позицію вручну в Desktop
git add <visual.json> && git rebase --continue

Підсумок: модульність PBIR ізолює конфлікт одним файлом.

Конфлікт звіту (PBIR)

18Обидва додали нову сторінку

Проблема: конфлікт у списку/порядку сторінок.

Причина: паралельне додавання сторінок.

# обєднати: лишити обидві сторінки, узгодити порядок
git add definition/.../pages && git commit

Підсумок: перевірити навігацію сторінок у Desktop.

Конфлікт звіту (PBIR)

19Несподіваний diff після авто-міграції на PBIR

Проблема: звіт дав великий diff формату після відкриття/збереження у Service.

Причина: автоматична міграція звіту на формат PBIR (за замовч. з 2026).

git add . && git commit -m "Migrate report to PBIR"  # окремим комітом

Підсумок: не змішувати міграцію з логічними змінами.

Конфлікт звіту (PBIR)

20Великий бінарний asset роздуває репо

Проблема: у звіт додано важке зображення; репо росте.

Причина: бінарні asset-и зберігаються повністю.

git lfs install
git lfs track "*.png"
git add .gitattributes && git commit -m "LFS for images"

Підсумок: великі бінарники — у Git LFS (окреме сховище).

Deploy / Fabric / CI

21Fabric workspace розходиться з гілкою

Проблема: у Fabric Git integration workspace показує uncommitted changes, що не збігаються з гілкою.

Причина: хтось правив прямо у workspace, оминаючи Git.

# джерело істини — Git. Або закомітити зміни workspace у гілку,
# або відкинути їх і зробити Update from Git у Fabric
git pull && git push

Підсумок: не правити workspace напряму, повз Git.

Deploy / Fabric / CI

22Deployment pipeline перезаписав міру з Service

Проблема: після deploy зникла міра, яку хтось додав напряму в Service.

Причина: джерелом деплою був Git, а зміну зробили повз нього.

git switch -c fix/restore-measure
git add tables/Sales.tmdl && git commit -m "restore measure"
git switch main && git merge fix/restore-measure && git push
# далі в Service: Dev → Update from Git → Deployment pipeline: Dev → Prod

Підсумок: усі зміни лише через Git → Dev (sync) → pipeline (Prod), не напряму в Service.

Deploy / Fabric / CI

23Втрата партицій incremental refresh після deploy

Проблема: після Git sync → pipeline → refresh історичні партиції зникли.

Причина: деплой метаданих перезаписав політику/партиції; дані не зберігаються в Git.

# деплоїти лише МЕТАДАНІ (не дані); перевірити RangeStart / RangeEnd
# після деплою — повний refresh для перебудови партицій
# діагностика партицій через XMLA: DAX Studio / Tabular Editor 2

Підсумок: партиції перебудувати після deploy; дані поза Git.

Deploy / Fabric / CI

24CI падає на Best Practice Analyzer

Проблема: PR не проходить — CI повертає помилки BPA (іменування мір тощо).

Причина: зміни порушують узгоджені правила якості.

# виправити локально: Tabular Editor → Best Practice Analyzer
git add . && git commit -m "fix BPA: measure naming"
git push                       # оновити PR → CI має пройти

Підсумок: дотримуватись узгоджених правил якості моделі.

Deploy / Fabric / CI

25Зміни в main є, але прод не оновився

Проблема: feature злито в main і запушено, але у Prod робочій області змін немає.

Причина: пропущено один із двох етапів: або Dev не синхронізовано з Git, або не виконано промоут Dev → Prod.

git log --oneline -1     # переконатись, що коміт справді в main/origin
# Power BI Service:
#   етап 1 — Dev workspace → Update from Git   (підтягнути main)
#   етап 2 — Deployment pipeline → Deploy: Dev → Prod

Підсумок: у прод зміни їдуть лише через ОБИДВА етапи — Git sync у Dev, потім pipeline у Prod.

Безпека

26Секрет / рядок підключення в M-виразі

Проблема: у Power Query (M) чи TMDL потрапив пароль; уже в історії на сервері.

Причина: чутливі дані закомічено в текст.

# 1) РОТУВАТИ секрет — він скомпрометований
# 2) видалити з історії:
git filter-repo --path definition/.../expressions.tmdl --invert-paths
git push --force-with-lease     # узгодити з командою
# 3) надалі — параметри / credential store

Підсумок: ніколи не зашивати секрети в код.

Налаштування

27Git щоразу питає логін і пароль

Проблема: кожен push/pull через HTTPS вимагає вводити креденшли заново.

Причина: не налаштований credential helper — Git нікуди не зберігає токен.

git config --global credential.helper manager   # Windows: Git Credential Manager
# macOS: credential.helper osxkeychain; Linux: store або cache
# альтернатива — перейти на SSH-ключі (урок «Автентифікація»)

Підсумок: токен вводиться один раз — далі його віддає менеджер креденшлів.

Продуктивність

28Клонування величезне й повільне

Проблема: репозиторій з довгою історією та великими asset-ами тягнеться дуже довго.

Причина: clone за замовчуванням забирає ВСЮ історію всіх файлів.

git clone --depth 1 <url>            # shallow: лише останній стан, без історії
git fetch --unshallow                # добрати повну історію пізніше, якщо треба
git sparse-checkout set Reports/PDP  # викачати в робочу папку лише потрібний шлях

Підсумок: shallow clone і sparse-checkout дають швидкий старт на великих репозиторіях.

Історія

29Помилковий merge у main уже запушено

Проблема: у main злили гілку, якої там бути не мало; історію переписувати не можна — main спільний.

Причина: merge зроблено не з тієї гілки або зарано.

git log --oneline --merges -3        # знайти SHA merge-коміту
git revert -m 1 <merge-sha>          # -m 1 = лишити сторону main, скасувати влиту гілку
git push

Підсумок: revert -m 1 безпечно скасовує merge новим комітом. Нюанс: щоб пізніше влити ту саму гілку знову, спершу revert самого revert-коміту.

Конфлікт звіту (PBIR)

30Після «безконфліктного» merge зник візуал

Проблема: merge пройшов без CONFLICT, але у звіті бракує візуала або він зламаний.

Причина: обидва правили сусідні блоки одного page.json; текстовий auto-merge склав синтаксично валідний, але логічно неповний JSON.

git log -p -- "*/pages/SalesPage/page.json"   # де саме загубився visualContainer
git checkout <sha> -- <шлях до page.json>     # повернути файл з коміту, де візуал ще був
# або відтворити візуал у Desktop і закомітити

Підсумок: «без конфлікту» ≠ «коректно». Після merge PBIP-звіт завжди відкривають у Desktop для валідації.

Модуль F — командна робота

Конвенції повідомлень комітів і назв гілок

Узгоджений формат робить історію читабельною й придатною для автоматизації.

Ціль урокуписати повідомлення й називати гілки за стандартом команди.
Conventional Commits
Стандарт повідомлень: <type>(<scope>): <subject>, за потреби — тіло й футер.
Простими словами: кожен коміт починається з типу зміни й короткого опису — легко читати й навіть будувати changelog автоматично.
feat(model): add YTD Sales measure
featнова функція (міра, таблиця, сторінка).
fixвиправлення помилки (формула, зв'язок).
refactorзміна без зміни поведінки (перейменування, чистка).
docsдокументація (README, описи).
choreслужбове (.gitignore, конфіги, оновлення).
# Добре:
feat(model): add Revenue YoY measure
fix(report): correct Sales page filter
# Погано:
update
fixes
wip
Атомарний коміт
Один коміт — одна логічна зміна.
Простими словами: не змішуй виправлення бага й нову міру в одному коміті — їх важко рев'ювити й відкочувати окремо.

Назви гілок

# Формат: <type>/<short-description>
feature/add-revenue-metric
fix/sales-filter
hotfix/prod-refresh-error
# Погано: my-branch, test, fix2
Зв'язок із задачами. Додавай ID робочого елемента в коміт чи PR: #123 або Closes #123 — Azure DevOps автоматично прив'яже зміну до задачі (трасування).
Модуль F — командна робота

Робочі процеси: Feature Branch, Gitflow, Trunk-based

Три поширені моделі. Для невеликої BI-команди важлива простота.

Ціль урокуобрати модель гілкування під розмір команди.

Feature Branch

Кожна зміна — у власній гілці від main, потім merge назад. Просто, ізольовано. Підходить більшості BI-команд.

Gitflow

Багато довгоживучих гілок (develop, release/*, hotfix/*, feature/*). Потужно, але складно й важко для PBIP-моделей.

Trunk-based

Усі часто зливають у одну гілку (main) маленькими порціями. Мінімум розходжень і конфліктів.

Чому простіше — краще для PBIP
PBIP/PBIR-діфи важко рев'ювити у PR, а довгі гілки накопичують складні merge-конфлікти моделі.
Простими словами: що менше й коротше гілки, то менше болю з конфліктами TMDL/JSON.
Модель цієї команди — близька до Feature Branch / Trunk-based: короткі feature-гілки → merge у main → push → Git sync публікує в робочу область (за потреби — deployment pipeline у Prod). Без важкого PR-review; гейтом якості є валідація в Desktop і CI/BPA (урок «Робочий процес: merge → деплой»).
Модуль F — командна робота

Azure DevOps: pipelines, задачі, політики

Що дає Azure DevOps навколо Git-репозиторію PBIP.

Ціль урокузвʼязати репозиторій із задачами та автоматичними перевірками.
Work items (робочі елементи)
Задачі / баги / User Stories в Azure DevOps з власними ID.
Простими словами: «квитки» роботи; згадка ID у коміті прив'язує зміну до задачі для трасування.
# у повідомленні коміту або описі:
fix(report): correct sales filter   #123
# Closes #123 / Fixes #456 — автозв'язок із задачею
Azure Pipelines (CI/CD)
Автоматичні конвеєри, що запускаються на події (push, оновлення гілки) і виконують кроки: валідація, тести, публікація.
Простими словами: «робот-складальник» на сервері. Для PBIP: перевірити модель (BPA), за потреби — автоматично опублікувати в Dev workspace.
# azure-pipelines.yml (спрощено)
trigger:
  branches: [ main ]
steps:
  - script: "запуск Best Practice Analyzer на .SemanticModel"   # CI-перевірка
  - script: "publish до Analytics-DEV"                        # авто-публікація в Dev
Branch policies (політики гілки)
Правила на захищеній гілці (напр. обов'язкова CI-перевірка перед merge).
Простими словами: необов'язковий запобіжник. Ця команда не вимагає рев'ю діфів, але може лишити автоматичну CI-перевірку як умову.
Узгоджено з процесом команди: Azure Pipelines автоматизує CI (BPA) і публікацію; якщо середовища розділені, промоут Dev → Prod іде через deployment pipeline; work items дають трасування. PR-review не є обов'язковим гейтом.
Модуль F — командна робота

Безпека та контроль доступу

Захист гілок, ролі та головне — секрети поза репозиторієм.

Ціль урокузахистити main і тримати секрети поза Git.
Branch protection (захист гілки)
Налаштування, що обмежують прямі зміни захищеної гілки (напр. лише з проходженням CI).
Простими словами: «замок» на main, щоб ніхто не пушив туди небезпечне напряму.
Рольова модель доступу
Різні рівні прав: read (перегляд), contribute (коміт у feature-гілки), maintainer/admin (керування).
Простими словами: кожен має рівно стільки прав, скільки треба для його ролі.

Секрети — НІКОЛИ не в Git

Патерн .env + .env.example
.env з реальними секретами додають у .gitignore; у репозиторій комітять лише .env.example — шаблон без значень.
Простими словами: у Git лежить «бланк» зі списком потрібних ключів, а самі ключі — лише локально.
# .env.example (у Git) — без значень
PBI_CLIENT_ID=
PBI_CLIENT_SECRET=
# .env (у .gitignore) — реальні значення, лише локально
Pre-commit hook для секретів
Локальний скрипт, що перед комітом шукає шаблони секретів і блокує коміт, якщо знайде.
Простими словами: «охоронець», що не дає випадково закомітити пароль.
Якщо секрет уже закомічено: 1) негайно ротувати (вважати скомпрометованим); 2) видалити з історії — git filter-repo або BFG Repo-Cleaner; 3) узгоджений push --force-with-lease. Деталі — кейс 26 у довіднику.
Модуль F — командна робота

Fabric Git integration: підключаємо робочу область

Фініш усього циклу курсу: merge у main → push → Update from Git. Тут розберемо, де саме в Power BI Service живе ця кнопка і що означають статуси поруч із нею. Це та частина процесу, яку аналітик робить у браузері, а не в терміналі.

Ціль урокупідключити робочу область до гілки main і впевнено читати статуси панелі Source control.
Fabric Git integration
Вбудований звʼязок робочої області Power BI / Fabric із гілкою Git-репозиторію: область уміє підтягувати коміти з гілки (Update from Git) і комітити зміни, зроблені в Service, назад у гілку.
Простими словами: «підписка» робочої області на гілку — натиснув Update, і область стала такою самою, як main.
Гілку main підключають до робочої області; після push у main достатньо натиснути Update from Git — і звіт опубліковано.

Передумови — дві, і обидві поза Git: робоча область має бути на ємності Fabric або Premium (на особистій My workspace чи звичайній Pro-області Git integration недоступна), а адміністратор tenant має дозволити Git integration у налаштуваннях (Admin portal → Tenant settings). Якщо чогось із цього немає — питання не до Git і не до тебе, а до адміністратора.

Підключення — робиться один раз

  1. Відкрий робочу область → Workspace settings → вкладка Git integration.
  2. Обери провайдера: Azure DevOps (або GitHub).
  3. Вкажи організацію, проєкт і репозиторій.
  4. Обери гілку (main) і папку в репозиторії, де лежить PBIP-проєкт.
  5. Натисни Connect — область підключено до гілки.

Панель Source control

Після підключення вгорі робочої області зʼявляється кнопка Source control з лічильником. Вона відкриває панель із двома вкладками: Updates — вхідні коміти з гілки, які ще не застосовані до області (саме тут кнопка Update from Git); Changes — зміни, зроблені безпосередньо в Service, які ще не закомічені в гілку.

Статуси елементів

SyncedЕлемент збігається з гілкою. Робити нічого не треба.
Update requiredГілка попереду: у main є новіший коміт, якого ще немає в області. Натисни Update from Git.
UncommittedЕлемент змінили прямо в Service, і ця зміна ще не потрапила в Git.
ConflictРозійшлись обидва боки: і в гілці новий коміт, і в Service — своя зміна того самого елемента.
Правити звіт прямо в Service небажано: джерело правди — Git. Якщо таке все ж сталося, одразу зроби Commit цієї зміни в гілку або Undo; інакше наступна синхронізація перетвориться на Conflict.

Як цей крок вбудовується у весь ланцюжок — наскрізний сценарій модуля 00: «Уся звʼязка разом». Типові проблеми синхронізації — кейси 21, 22 і 25 у довіднику проблем.

Модуль F — командна робота

Реальний процес: merge у main → деплой у прод

Огляд PBIP/PBIR-діфів у PR/веб-інтерфейсі непрактичний (текстовий TMDL і JSON-шум, без візуального прев'ю звіту), тому команда не робить ставку на PR-review. Зміни швидко зливаються в main, а справжній контроль — це валідація перед злиттям, CI і контрольована публікація через Git sync (за потреби — з окремим Prod через deployment pipeline).

Ціль урокупройти повний шлях зміни: гілка → main → робоча область (та опційно Prod).
Звичайна публікація (базовий варіант). Одна робоча область, підключена до main: merge → push → Update from Git — і звіт уже у користувачів. Схема нижче показує розширений варіант з окремими Dev і Prod; якщо у твоєї команди одне середовище — етап 2 просто відсутній.

гілка

робота + перевірка в Desktop
→
гілка
main
merge + git push у веб (origin)
→
workspace
Dev
Update from Git

workspace

етап 2 · опційно
  1. Робота у feature/*; перед злиттям проєкт відкривають у Power BI Desktop і перевіряють (модель валідна, міри рахують).
  2. feature-гілку зливають у main і пушать у веб (Azure DevOps).
  3. Етап 1 — Git sync: main під'єднано до робочої області через Git integration. У Power BI Service роблять Update from Git — область отримує зміни. Якщо середовище одне, публікація на цьому завершена.
  4. Етап 2 (опційний) — Deployment pipeline: команди з окремими Dev і Prod промоутять перевірений вміст із Dev у Prod.

Команди

git switch feature/okr-summary
# … правки в TMDL/PBIR, перевірка у Power BI Desktop …
git switch main
git merge feature/okr-summary    # локальне злиття у main
git push                          # відправити у веб (origin)
# далі — не в Git, а в Power BI Service:
#   етап 1) робоча область → Update from Git   (звіт опубліковано)
#   етап 2, якщо є Dev/Prod) Deployment pipeline: Dev → Prod
Без термінала. Git-частину цих кроків можна пройти кнопками у Sourcetree: переглянути diff перед комітом, злити feature-гілку в main і зробити Push. Перевірка проєкту перед злиттям — як і раніше, у Power BI Desktop.
Замість PR-review бар'єрами якості є: перевірка моделі в Desktop перед merge, автоматичний CI/BPA на .SemanticModel, і поетапний деплой — у Prod зміни потрапляють лише після проходження Dev.

Чим це відрізняється від класичного PR-флоу

Класичнийfeature → PR з review та обговоренням діфів → merge у main.
Тутfeature → merge у main (без важкого review діфів) → push → Git sync публікує в робочу область (опційно далі pipeline у Prod).
Hotfix: терміновий фікс так само зливають у main і пушать; Git sync публікує його в робочу область (у схемі з Dev/Prod — далі промоут через pipeline). За потреби — cherry-pick потрібного коміту (Сценарій 8).

Підсумок курсу

  • Зони: working dir → staging (add) → repo (commit) → remote (push).
  • Локально vs хмара: дві окремі копії; fetch/pull забирають, push відправляє; при розбіжності — pull --rebase.
  • Гілки: merge зберігає історію, rebase переписує лінійно.
  • Виправлення: restore (файли), reset (локально), revert (безпечно для спільних), reflog (рятівне коло).
  • PBIP: текст замість бінарника → реальний контроль версій; TMDL (модель), PBIR (звіт), .pbi/ → .gitignore.
  • Публікація: main → робоча область (Git sync) — це базовий шлях; окремий Prod через deployment pipeline — опційний другий етап. Прод не є гілкою Git, це робоча область.
Золоте правило: переписуй історію (amend, reset, rebase) лише поки коміти локальні. У main зміни їдуть обережно — через merge і push, а до користувачів — через контрольовану публікацію з main.
Модуль F — командна робота

Дані після публікації: refresh, credentials, gateway

Синхронізація пройшла, а звіт порожній або скаржиться на credentials? Це найтиповіша точка паніки новачка. Спокійно: це не поломка, а очікуваний крок процесу.

Ціль урокурозуміти, що Git привозить лише визначення, а дані наповнюються через refresh — і вміти пройти кроки credentials → gateway → розклад.

Ментальна модель: Git привозить «креслення» — визначення моделі та звіту (TMDL і PBIR). Самих ДАНИХ у репозиторії немає і бути не може: таблиці наповнюються вже в Power BI Service, коли датасет виконує refresh і сам іде в джерела. Тому одразу після першої синхронізації звіт цілком законно порожній.

Після першої синхронізації датасету

  1. Налаштування датасету: Settings → Semantic models → Data source credentials — задай облікові дані кожного джерела (Edit credentials).
  2. Якщо джерело on-prem — наприклад, SQL Server у локальній мережі компанії — привʼяжи датасет до gateway (див. нижче).
  3. Запусти refresh разово (Refresh now) і переконайся, що він пройшов без помилок; потім налаштуй розклад (Scheduled refresh).
Gateway (on-premises data gateway)
Службова програма-міст між хмарою Power BI Service і джерелами даних у локальній мережі: хмара не бачить локальний сервер напряму, тож звертається до gateway, а той уже ходить у джерело.
Простими словами: консьєрж, який має ключ від будинку — хмара сама в будинок не заходить, а просить консьєржа винести дані за запитом.
Датасет із джерелом SQL Server у локальній мережі: без привʼязки до gateway refresh у Service падатиме, хоча в Power BI Desktop усе працювало.

Якщо середовища розділені (Dev і Prod — окремі робочі області), підключення до джерел розводять параметрами моделі (наприклад, параметр із назвою сервера) та deployment rules у deployment pipeline: під час промоуту Dev → Prod правило підставляє продові значення. Це опційний рівень — якщо середовище одне, цей абзац можна пропустити.

Помилка «credentials required» після синхронізації — не «Git щось зламав», а штатний крок публікації: креслення приїхали, тепер дай Service ключі від джерел. Суміжний випадок із партиціями incremental refresh — кейс 23 у довіднику проблем.

Модуль F — командна робота

Практикум: від .pbix до опублікованого звіту

Єдиний урок курсу, де все робиться руками на твоїй машині: від тестового .pbix до опублікованого звіту. Виділи 30–60 хвилин і проходь кроки по черзі — після кожного є чекпойнт самоперевірки.

Ціль урокупройти повний цикл власноруч: .pbix → PBIP → git init → віддалений репозиторій → feature-гілка → merge → (опційно) публікація через Update from Git.
  • 1. Save As → PBIP. Візьми тестовий звіт (.pbix, НЕ робочий) і в Power BI Desktop зроби Save As у формат PBIP у нову порожню папку. ✅ Перевір: поруч зʼявились папки .Report і .SemanticModel та файл .gitignore.
  • 2. git init. Відкрий Git Bash у цій папці й виконай git init, потім git status. ✅ Перевір: cache.abf НЕ у списку файлів — отже, .gitignore працює (якщо він там є — повернись до уроку «.gitignore для PBIP»).
  • 3. Перший коміт. git add . і git commit -m "feat: initial PBIP". ✅ Перевір: git log --oneline показує рівно один коміт.
  • 4. Віддалений репозиторій. Створи приватний репозиторій в Azure DevOps або GitHub, підключи його: git remote add origin …, потім git push -u origin main. ✅ Перевір: файли проєкту видно у веб-інтерфейсі репозиторію.
  • 5. Feature-гілка. git switch -c feature/test: зміни одну міру в Power BI Desktop, збережи проєкт і подивись diff у Sourcetree чи VS Code. ✅ Перевір: бачиш свою зміну в tables/*.tmdl і відрізняєш її від технічного шуму (урок «Читаємо diff PBIP»).
  • 6. Merge у main. Закоміть зміну в гілці, потім git switch main, git merge feature/test, git push. ✅ Перевір: у веб-інтерфейсі main містить обидва коміти.
  • 7. (Опційно) Публікація. Якщо маєш доступ до Fabric-ємності — підключи ТЕСТОВУ робочу область до main (урок «Fabric Git integration») → Update from Git → задай credentials і запусти refresh (урок «Дані після публікації»). ✅ Перевір: звіт відкривається й показує дані.
Щось пішло не так? Не панікуй: майже кожну ситуацію вже розібрано в довіднику проблем, а команди виправлення — у модулі 04.
