Відкат незакомічених змін

Поки зміна не в коміті, її можна просто відкинути — головне знати, з якої зони ти її прибираєш.

Цільскасовувати те, що ще не в історії: зіпсований файл і зайвий файл у staging
Я поруч, і на цій сторінці можна не боятися: тут саме те місце, де можна безкарно ламати — тож ламай сміливо. Застрягнеш — тисни Підказку, це не соромно.

Коротке нагадування: зміна проходить три зони — робоча директорія → staging (git add) → історія (git commit). Поки коміту не було, скасування — це не «переписування історії», а звичайне прибирання: git restore <файл> повертає файл до останньої закоміченої версії, а git restore --staged <файл> лише виймає файл зі staging, не чіпаючи його вміст на диску.

Для Power BI це щоденні ситуації: покрутив формулу міри в definition/model.tmdl, зберіг PBIP — і зрозумів, що стало гірше. Або машинально зробив git add . і в staging заліз файл, якому в репозиторії взагалі не місце.

Тут особливо важливо: у тренажері restore нічого реально не видаляє, це безпечна пісочниця. А от справжню руку тренуй у тестовій папці, а не в робочому проєкті — навмисно зіпсуй там файл і поверни його тими самими командами, поки не звикнеш робити це впевнено й без страху за реальні дані.

Перша задача: ти експериментував із мірою, коміту ще не було, і треба повернути definition/model.tmdl до останньої закоміченої версії.

restore незворотний для незакомічених правок. Git повертає файл з останнього коміту, а твої нові зміни ніде не зберігаються — їх не дістане навіть reflog. Перед git restore переконайся, що ці правки тобі справді не потрібні.

Друга задача: у staging випадково потрапив cache.abf — кеш даних, який не комітять ніколи. Прибери його зі staging так, щоб файл лишився на диску. А щоб таке не повторювалося, кеш даних узагалі має сидіти в .gitignore.

Скасувати коміт: reset для своїх, revert для спільних

Одне питання визначає інструмент: чи бачив цей коміт хтось, крім тебе?

Цільвибирати правильний інструмент: reset --soft для незапушеного коміту, revert — для запушеного

Коли зміна вже в історії, скасування працює інакше. git reset пересуває вказівник гілки назад — коміт зникає з історії, а зміни (у режимі --soft) повертаються у staging. git revert нічого не стирає: він додає новий коміт, який скасовує зміни поганого. Теорію з усіма трьома режимами reset розібрано в модулі 04.

Перший сценарій: ти закомітив зміну моделі з повідомленням «wip» — зарано і з поганим описом. Коміт іще не запушений, тож його можна тихо скасувати, зберігши зміни у staging для нормального коміту.

Ключове правило вибору: чи бачив цей коміт хтось інший? Якщо він тільки локальний і на сервер не потрапив — можна reset, ти переписуєш лише власну чернетку. Якщо коміт уже запушено — тільки revert: переписану історію сервер відхилить, а колеги, які встигли її забрати, отримають головний біль.

Другий сценарій: коміт із заміною формули маржі зламав міру і вже на сервері. Історію переписувати не можна — зроби коміт-скасування і відправ його на сервер.

Stash посеред роботи і схема вибору reset/revert

Термінові фікси не питають, чи зручно тобі зараз — stash дає змогу відкласти незавершене без коміту-огризка.

Цільховати незавершену роботу в stash заради термінового фіксу і впевнено вибирати між reset і revert

git stash — це «кишеня»: він забирає незакомічені зміни з робочої директорії і складає їх окремо, лишаючи файли чистими. Потім git stash pop повертає все як було. Класична причина — треба терміново перемкнутися на іншу гілку, а комітити напівготову роботу не хочеться.

Сценарій — найдовший у практикумі: ти посеред роботи у feature/kpi-cards із незакоміченими правками, і тут прилітає терміновий фікс міри в main. Спробуй спершу одразу git switch main — Git не пустить, бо змінений файл відрізняється між гілками і перемикання перезаписало б твої незбережені правки (саме в такому разі Git блокує switch; якби файл між гілками не відрізнявся — Git спокійно перемкнув би гілку й переніс правки з собою). Далі: сховати → перемкнутися → зробити фікс → повернутися → дістати зі stash.

Порада: stash — для незакомічених змін і на короткий час. Якщо робота відкладається на дні — краще закоміть її у своїй feature-гілці: коміт видно в історії, а про stash легко забути.

І на закріплення — схема вибору між reset і revert. Прокрути три кроки: поганий коміт; що робить із графом reset --hard; що робить revert.

Reset проти revert

Тепер перевір себе на робочому сценарії. Памʼятай наш командний потік: feature-гілка → merge у main → push → Git sync публікує main у робочу область — тобто main завжди спільна.

Екзамен: скасування і порятунок

Три питання без підказок: reflog, режими reset і amend запушеного коміту.

Цільчитати reflog, знати долю змін у кожному режимі reset і наслідки amend запушеного коміту

Останній рубіж цієї частини. Згадай: git reflog — журнал усіх переміщень HEAD, рятівний трос після випадкового reset --hard; три режими reset відрізняються долею змін (staging → робоча директорія → ніде); а --amend не «редагує» коміт, а створює новий замість старого.
