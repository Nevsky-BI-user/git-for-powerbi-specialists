Модуль A — основи

Навіщо потрібен Git

Почнемо з найпростішого, не припускаючи жодних попередніх знань.

Ціль урокуназвати проблеми, які Git вирішує для Power BI-розробника.

Що таке «версія» файлу

Коли ти зберігаєш файл, на диску лишається його поточний стан. Якщо завтра щось зміниш і збережеш знову — старий стан зникне назавжди. Версія — це збережений знімок файлу в певний момент. Контроль версій дозволяє мати багато таких знімків і повертатись до будь-якого.

Система контролю версій (Version Control System, VCS)
Програма, що запам'ятовує кожен збережений стан проєкту й дозволяє повертатись назад, порівнювати стани й об'єднувати правки кількох людей.
Простими словами: як «точки збереження» (save points) у відеогрі. У будь-який момент можна повернутись до раніше збереженої точки, навіть якщо після неї все пішло не так.
Приклад: ти зламав звіт правкою. З Git відкочуєшся до вчорашнього стану однією командою.
Git
Найпоширеніша система контролю версій. Працює локально на твоєму комп'ютері: уся історія зберігається у твоїй папці проєкту.
Простими словами: «розумний журнал змін», що живе прямо в папці проєкту й пам'ятає кожен твій крок.

Проблема без Git

Ручний підхід: report_final.pbix, report_final_v2.pbix, report_final_REALLY_final.pbix. Неможливо побачити різницю, безпечно об'єднати правки двох людей чи зрозуміти, що зламалось і коли.

Git ≠ GitHub ≠ Sourcetree

Ці три речі часто плутають. Вони різні:

Git

Сама програма контролю версій. Працює локально, без інтернету.

Як двигун автомобіля.

GitHub / Azure DevOps

Хмарні сервіси, що зберігають копію репозиторію онлайн для спільної роботи та резервування.

Як гараж, де машину бачить уся команда.

Sourcetree

Графічна програма (GUI) з кнопками замість команд. Кожна кнопка викликає Git під капотом.

Як зручна панель приладів над двигуном.
Як це виглядає у щоденній роботі. Відкриваєш звіт у Power BI Desktop → зберігаєш → переглядаєш зміни в Sourcetree → комітиш → пушиш у GitHub чи Azure DevOps, а якщо виник конфлікт — вирішуєш його в merge-редакторі VS Code. Усі ці інструменти ти вже налаштував у модулі 00 «Старт та інструменти».
Для Power BI це критично. Бінарний .pbix Git бачить як один нероздільний блок. Формат PBIP (текст + JSON + TMDL) робить кожну зміну DAX-міри чи візуала видимою. Цьому присвячені модулі E–F.
Модуль A — основи

Базові поняття

Сім термінів-фундаментів. Кожен пояснено так, ніби ти бачиш Git уперше.

Ціль урокубез плутанини розуміти слова репозиторій, коміт, гілка, remote.
Repository (репозиторій, скорочено «репо»)
Папка проєкту, за якою стежить Git. Усередині неї Git створює приховану підпапку .git, де зберігає всю історію.
Простими словами: звичайна папка проєкту + невидимий «чорний ящик» .git, що пам'ятає кожну зміну.
Напр. папка C:\Repos\PDP.
Commit (коміт)
Збережений знімок усіх файлів у певний момент разом із метаданими: хто, коли, з яким описом. Кожен коміт має унікальний номер — SHA.
Простими словами: «фотографія» проєкту з підписом. Серія фотографій = історія.
Коміт «Додано міру Total Sales» зберігає всі файли саме в той момент.
SHA (хеш коміту)
40-символьний код, обчислений із вмісту коміту. Унікальний «паспорт» коміту. Зазвичай пишуть перші 7 символів.
Простими словами: як штрих-код фотографії — двох однакових не буває.
a1b2c3d
Working Directory (робоча директорія)
Файли проєкту, які ти зараз бачиш на диску й редагуєш у редакторі.
Простими словами: те, що зараз «на столі» — поточний стан файлів.
Staging Area / Index (зона підготовки)
Проміжна «коробка». Командою git add ти кладеш туди зміни, які увійдуть у наступний коміт.
Простими словами: як кошик у магазині. Спершу складаєш товари (add), потім оплачуєш усе разом (commit). У Sourcetree цей кошик — список Staged files, у VS Code — Staged Changes.
Змінив 5 файлів, але хочеш закомітити лише 2 пов'язані — кладеш у кошик саме їх.
Branch (гілка)
Рухомий вказівник на коміт; окрема лінія розробки.
Простими словами: як паралельний всесвіт проєкту. Можеш експериментувати в окремій гілці, не чіпаючи основну.
main — основна; feature/svg-badge — під нову фічу.
HEAD
Вказівник на те, де ти зараз «стоїш» — на яку гілку чи коміт.
Простими словами: як «ви тут» на карті торгового центру.
HEAD → main → a1b2c3d
Remote (віддалений репозиторій)
Копія репозиторію на сервері (GitHub, Azure DevOps). Стандартна назва — origin.
Простими словами: спільна хмарна копія, до якої під'єднуються всі члени команди.
Запис HEAD~. HEAD~1 — батько поточного коміту (на 1 назад), HEAD~2 — на 2 назад. Зручно адресувати: git reset HEAD~2 = «відкотись на 2 коміти».
Модуль A — основи

Як Git зберігає дані: знімки, не дельти

Розуміння цього пояснює, чому Git швидкий і чому гілки в ньому майже безкоштовні.

Ціль урокупояснити модель знімків і чому гілки в Git дешеві.
Дельта (delta)
Збережена РІЗНИЦЯ між версіями файлу: базова версія + список змін.
Простими словами: як список правок «у рядку 5 додай слово». Щоб дістати стару версію, треба застосувати всі правки по черзі.
Знімок (snapshot)
Повний стан усіх файлів на момент коміту. Якщо файл не змінився — Git зберігає лише посилання на попередню копію.
Простими словами: як повне фото проєкту щоразу; незмінні файли — це «те саме фото, дивись туди».

Старі системи (дельти)

Базовий файл + ланцюг різниць. Щоб дістати версію — застосувати всі дельти послідовно. Економить місце, але повільніше.

Git (знімки)

Повний знімок на кожен коміт; незмінні файли — посилання. Порівняння й перемикання дуже швидкі. Сховище стискається у pack-файли.

# Дельти: версія = база + delta + delta + delta …
# Git: кожен коміт — знімок; file2 не змінився → посилання на попередній
Коміт 1: [file1: A]  [file2: B]
Коміт 2: [file1: A'] [file2: → з Коміту 1]
SHA / хеш як адреса
Кожен об'єкт (коміт, файл, дерево) адресується SHA-хешем свого вмісту.
Простими словами: однаковий вміст → однаковий хеш; зміна вмісту → інший хеш. Тому Git одразу «бачить», що саме змінилось.
Чому це важливо для PBIP. Незмінені .tmdl-файли не дублюються в історії — змінюється лише те, що ти справді правив. Тому історія моделі лишається компактною й читабельною.
Модуль A — основи

Три зони та життєвий цикл файлу

Файл у Git завжди в одній із зон. Розберемо кожну, потім подивимось рух наживо.

Ціль урокупростежити шлях зміни: робоча папка → staging → коміт.
Чотири місця, де «живе» файл
1) Working Directory — на диску, ти редагуєш. 2) Staging — у кошику, готовий до коміту. 3) Repository (.git) — закомічено, у локальній історії. 4) Remote — на сервері, доступно команді.
Простими словами: чернетка на столі → кошик у магазині → куплено й удома → відправлено друзям.
Як ці зони виглядають у графічних інструментах. Зберіг звіт у Power BI Desktop (у форматі PBIP) — і змінені файли одразу з'явились у Sourcetree в списку Unstaged files: це і є Working Directory. Натиснув Stage — файл перейшов у список Staged files, тобто git add однією кнопкою. У VS Code ті самі списки звуться Changes та Staged Changes, а «+» біля файлу — той самий git add.
Інтерактив: рух файлу командами Git
редагуєшWorking Dir
кошикStaging
локальноRepository
серверRemote
Файл змінено в робочій директорії — Git бачить його як «modified».

Стандартний цикл

git status                 # що змінилось і в якій зоні
git add model.tmdl        # покласти у staging (кошик)
git commit -m "Додано міру Salary Range Position"  # зафіксувати
git push                   # відправити на сервер
Навіщо аж дві зони (staging + commit)? Кошик дозволяє зібрати коміт вибірково: змінив багато — закомітив лише пов'язане між собою. Так історія лишається логічною й читабельною.

Спробуй сам: перші команди руками

Ці команди виконуються у тій самій папці проєкту, куди ти перейшов через cd (модуль 01) — Git завжди дивиться на поточну робочу папку.
